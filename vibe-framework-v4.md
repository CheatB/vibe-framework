## 1. Философия: Мозг + Руки

### Ключевая идея

Фреймворк разделяет **что делать** и **как делать**:

| | Мозг (Фреймворк) | Руки (Инструменты) |
|---|---|---|
| **Где живёт** | `CLAUDE.md`, `rules/`, workflow-маркеры | hooks, agents, plugins, skills, commands |
| **Что делает** | Описывает ЧТО проверять, КОГДА запускать, КАКОЙ режим выбрать | Выполняют конкретную работу: сканируют, форматируют, ревьюят |
| **Пример** | «Проверь безопасность при каждом коммите» | `hooks/scripts/security-scan.sh` |
| **Пример** | «Сделай код-ревью после >20 строк» | `agents/code-reviewer.md` |

### Зачем это нужно

Чистый промпт-инжиниринг — это когда ты пишешь простыню текста в CLAUDE.md и надеешься, что модель всё запомнит и правильно сделает. Проблемы:

- Контекстное окно конечно — промпт на 10к токенов жрёт место для полезной работы
- Модель «забывает» правила в длинных сессиях
- Нет гарантий выполнения — модель может проигнорировать инструкцию
- Нет автоматизации — всё ручное

**Мозг + Руки** решает это:

- Правила минимальны — ссылаются на конкретные инструменты
- Hooks работают автоматически — не зависят от «настроения» модели
- Агенты подключаются по требованию — не забивают контекст постоянно
- Plugins дают доступ к реальным данным — LSP, grep, файловая система

### Что выбрать: Opus vs Sonnet

| Модель | Когда использовать | Стоимость |
|---|---|---|
| **Opus** | Архитектура, планирование, сложная логика, ревью | ~5x дороже Sonnet |
| **Sonnet** | Рутинная реализация, тесты, субагенты | Базовая |
| **Haiku** | Простые правки, форматирование, быстрые проверки | ~10x дешевле Sonnet |

**Правило:** Opus думает, Sonnet пишет, Haiku проверяет.

## 2. Четыре уровня работы

Каждая задача попадает в один из четырёх уровней. Уровень определяет глубину процесса, набор инструментов и количество проверок.

```
┌─────────────────────────────────────────────────┐
│ 🟣 ПРОЕКТ — создание с нуля (дни/недели)        │
│  ┌──────────────────────────────────────────┐    │
│  │ 🔵 ЭПИК — крупная функциональность (часы)│    │
│  │  ┌───────────────────────────────────┐   │    │
│  │  │ 🟢 ФИЧА — одна возможность (мин)  │   │    │
│  │  │  ┌────────────────────────────┐   │   │    │
│  │  │  │ ⚡ ФИКС — быстрый патч     │   │   │    │
│  │  │  └────────────────────────────┘   │   │    │
│  │  └───────────────────────────────────┘   │    │
│  └──────────────────────────────────────────┘    │
└─────────────────────────────────────────────────┘
```

| Уровень | Время | Ключевые фазы | Агенты |
|---|---|---|---|
| **Проект** | Дни | Инфра → спецификация → архитектура → декомпозиция → реализация → тесты → деплой | Все |
| **Эпик** | Часы | Планирование → wave-параллелизм → интеграция → QA | PM + команда |
| **Фича** | 15-60 мин | Brainstorm → план → реализация блоками → тесты → коммит | 3-5 агентов |
| **Фикс** | 5-15 мин | Определи → исправь → проверь | 0-1 агент |

## 3. Уровень: Проект — от нуля до работающего продукта

Весь процесс запускается **одной командой** и идёт непрерывным потоком, перетекая из фазы в фазу.

**Команда:** `/new-project`

Эта команда запускает полный пайплайн из 7 фаз:

```
/new-project
    │
    ├── Фаза 0: Подготовка инфраструктуры (VPS)
    ├── Фаза 1: Пользовательская спецификация (User Spec)
    ├── Фаза 1.8: Бизнес-анализ [NEW v4.1]
    │   ├── 1.8.1 Legal Compliance (РФ)
    │   ├── 1.8.2 Monetization
    │   ├── 1.8.3 GTM (Go-to-Market)
    │   ├── 1.8.4 Analytics
    │   └── 1.8.5 Accessibility
    ├── Фаза 2: Техническая спецификация (Tech Spec)
    ├── Фаза 3: Декомпозиция на задачи
    ├── Фаза 4: Реализация
    ├── Фаза 5: Тестирование
    └── Фаза 6: Финализация (QA + Deploy + Docs)
```

Между фазами нет ручного переключения. ИИ сам определяет, когда текущая фаза завершена, показывает результат пользователю для подтверждения и переходит к следующей. Пользователь в любой момент может остановить процесс, скорректировать курс или откатиться к предыдущей фазе.

### Фаза 0: Подготовка инфраструктуры (VPS)

**Цель:** Поднять и настроить серверное окружение для нового проекта.

> 💡 **Когда пропустить:** Если VPS уже настроен и ты добавляешь проект на существующий сервер — Фаза 0 сводится к созданию директории проекта и docker-compose.

#### Шаг 1: Создание и базовая настройка VPS

**Подключение к VPS (через MCP SSH или напрямую):**

```bash
# Подключение к серверу
ssh root@<IP_ADDRESS>

# Обновление системы
apt update && apt upgrade -y

# Установка базовых пакетов
apt install -y curl wget git htop nano ufw fail2ban
```

**Настройка безопасности:**

```bash
# Создание рабочего пользователя (не root!)
adduser deploy
usermod -aG sudo deploy

# Настройка SSH
# Отключить вход по паролю, только по ключу
sed -i 's/#PasswordAuthentication yes/PasswordAuthentication no/' /etc/ssh/sshd_config
sed -i 's/PermitRootLogin yes/PermitRootLogin no/' /etc/ssh/sshd_config
systemctl restart sshd

# Настройка файрвола
ufw default deny incoming
ufw default allow outgoing
ufw allow ssh
ufw allow 80/tcp
ufw allow 443/tcp
ufw enable

# Настройка fail2ban
systemctl enable fail2ban
systemctl start fail2ban
```

#### Шаг 2: Установка Docker и docker-compose

```bash
# Установка Docker
curl -fsSL https://get.docker.com | sh
usermod -aG docker deploy

# Установка docker-compose (v2)
apt install -y docker-compose-plugin

# Проверка
docker --version
docker compose version
```

#### Шаг 3: Настройка Nginx + SSL (Reverse Proxy)

```bash
# Установка Nginx
apt install -y nginx certbot python3-certbot-nginx

# Базовый конфиг для проекта (шаблон)
cat > /etc/nginx/sites-available/project-name << 'EOF'
server {
    listen 80;
    server_name project.example.com;

    location / {
        proxy_pass http://localhost:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
EOF

# Включение сайта
ln -s /etc/nginx/sites-available/project-name /etc/nginx/sites-enabled/
nginx -t && systemctl reload nginx

# SSL-сертификат (после настройки DNS!)
certbot --nginx -d project.example.com
```

#### Шаг 4: Настройка базы данных и Redis

```bash
# PostgreSQL через Docker (рекомендуемый способ)
# Будет в docker-compose проекта

# Redis через Docker (рекомендуемый способ)
# Будет в docker-compose проекта

# Если нужны standalone-инстансы:
apt install -y postgresql redis-server
systemctl enable postgresql redis-server
```

#### Шаг 5: Настройка CI/CD (GitHub Actions → VPS)

```bash
# На VPS: настройка для автодеплоя
mkdir -p /home/deploy/projects/project-name
cd /home/deploy/projects/project-name

# SSH-ключ для GitHub Actions
ssh-keygen -t ed25519 -f /home/deploy/.ssh/github_actions -N ""
cat /home/deploy/.ssh/github_actions.pub >> /home/deploy/.ssh/authorized_keys
```

**GitHub Actions workflow (шаблон):**
```yaml
# .github/workflows/deploy.yml
name: Deploy
on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Deploy to VPS
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.VPS_HOST }}
          username: deploy
          key: ${{ secrets.VPS_SSH_KEY }}
          script: |
            cd /home/deploy/projects/project-name
            git pull
            docker compose up -d --build
```

#### Шаг 6: Мониторинг и логирование

```bash
# Логирование Docker-контейнеров
# docker-compose настроен на json-file с rotation

# Простой мониторинг (для начала)
# Healthcheck в docker-compose
# Логи через: docker compose logs -f

# Продвинутый мониторинг (по необходимости):
# - Grafana + Prometheus
# - Sentry для ошибок
# - Uptime Kuma для мониторинга доступности
```

#### Шаг 7: Создание структуры проекта на VPS

```bash
# Инициализация директории
cd /home/deploy/projects/project-name
git init
git remote add origin git@github.com:CheatB/project-name.git

# Создание docker-compose.yml (базовый шаблон)
cat > docker-compose.yml << 'EOF'
version: '3.8'

services:
  app:
    build: .
    restart: unless-stopped
    ports:
      - "8000:8000"
    env_file:
      - .env
    depends_on:
      - db
      - redis

  db:
    image: postgres:16-alpine
    restart: unless-stopped
    volumes:
      - pgdata:/var/lib/postgresql/data
    env_file:
      - .env

  redis:
    image: redis:7-alpine
    restart: unless-stopped
    volumes:
      - redisdata:/data

volumes:
  pgdata:
  redisdata:
EOF
```

- [ ] Docker + docker-compose установлены
- [ ] Nginx настроен как reverse proxy

### Фаза 1: Пользовательская спецификация (User Spec)

**Цель:** Понять, ЧТО строим, ДЛЯ КОГО и ЗАЧЕМ — максимально глубоко и детально.

#### Подключение субагента: Бизнес-аналитик

На эту фазу ИИ запускает субагент **`business-analyst`** — въедливого и вдумчивого бизнес-аналитика. Этот агент:

- Задаёт **максимально большое количество вопросов** для выяснения глубокого контекста
- Не стесняется переспрашивать и уточнять
- Задаёт вопросы батчами по 3-5 штук (чтобы не перегружать, но и не упустить)
- Если пользователь не может ответить на вопрос → агент **предлагает свои варианты ответов** на выбор ИЛИ **кладёт вопрос в Бэклог** проекта для проработки позже
- Не удовлетворяется поверхностными ответами — копает вглубь

#### Процесс глубинного интервью:

**Цикл 1 — Общее понимание проекта:**
- Какую проблему решаем? Кто страдает от этой проблемы прямо сейчас?
- Кто целевая аудитория? Можешь описать типичного пользователя?
- Какие основные фичи нужны? Что ОБЯЗАТЕЛЬНО должно быть в первой версии?
- Есть ли аналоги/конкуренты? Чем мы хотим отличаться?
- Какая бизнес-модель? Как будем зарабатывать (если коммерческий)?
- Какой масштаб ожидаем? 10 пользователей? 1000? 100000?

**Цикл 2 — Пользовательские сценарии (User Stories):**
- Что делает пользователь в первую очередь после открытия/запуска?
- Какие основные пользовательские пути (happy path)?
- Какие действия критичны и не должны ломаться ни при каких обстоятельствах?
- Есть ли разные роли пользователей? Какие у каждой права?
- Как пользователь узнаёт о продукте? Как регистрируется?
- Какие данные пользователь вводит? Какие получает?

**Цикл 3 — Интерфейс и UX:**
- Какой тип интерфейса: web, mobile, desktop, Telegram-бот, CLI?
- Есть ли референсы дизайна (сайты/приложения, которые нравятся)?
- Какие экраны/страницы нужны?
- Как выглядит навигация?
- Поддержка мобильных устройств? Адаптивность?
- Тёмная тема? Мультиязычность?
- Доступность (a11y): нужно ли учитывать?

**Цикл 4 — Данные и интеграции:**
- Какие данные хранятся? Какие сущности?
- Есть ли внешние источники данных?
- С какими сервисами нужна интеграция? (платежи, email, SMS, AI, maps и т.д.)
- Нужен ли импорт/экспорт данных?
- Какие форматы данных? (JSON, CSV, Excel, PDF)
- Нужна ли аналитика/отчёты?
- Как обрабатываем файлы? Есть ли лимиты по размеру?

**Цикл 5 — Нефункциональные требования:**
- Какое допустимое время отклика? (< 200ms? < 1s? < 5s?)
- Какой аптайм нужен? (99%? 99.9%?)
- Какие требования по безопасности? (GDPR, шифрование, 2FA)
- Нужна ли работа в офлайне?
- Какой бюджет на инфраструктуру? (VPS, облако, free tier)
- Есть ли требования по SEO?
- Бэкапы: как часто, куда, сколько хранить?

**Цикл 6 — Edge-cases и обработка ошибок:**
- Что если у пользователя нет интернета?
- Что если данные повреждены?
- Что если пользователь пытается ввести невалидные данные?
- Какие ошибки показываем пользователю? На каком языке?
- Что если внешний сервис (API) недоступен?
- Как обрабатываем одновременный доступ (конкурентность)?
- Что если пользователь забыл пароль? Потерял доступ?

**Цикл 7 — Деплой и операционка:**
- Где хостим: VPS Beget / облако / другое?
- Нужен ли Docker?
- Какие домены/поддомены?
- Как будем обновлять? Zero-downtime или окно обслуживания?
- Нужен ли staging-окружение?
- Кто будет поддерживать продукт после запуска?

#### Работа с неотвеченными вопросами:

Когда пользователь не может ответить на вопрос, бизнес-аналитик:

1. **Предлагает варианты ответов** на основе типовых решений:
   ```
   Вопрос: Как будем обрабатывать оплату?

   Ты пока не определился? Вот типовые варианты:
   A) ЮKassa — популярна в РФ, комиссия 3.5%
   B) Stripe — международный, удобный API, комиссия 2.9%+30¢
   C) Telegram Payments — если бот, встроенная оплата
   D) Пока без оплаты — добавим позже

   Выбери или опиши свой вариант.
   ```

2. **Кладёт в Бэклог**, если вопрос вообще не актуален сейчас:
   ```
   Ок, вопрос по мультиязычности пока неактуален.
   Кладу в бэклог: [BACKLOG] Определить стратегию мультиязычности
   Вернёмся к нему позже.
   ```

#### Проверка полноты:

Запускаются валидаторы параллельно:

| Валидатор | Что проверяет |
|---|---|
| `userspec-quality-validator` | Структура документа, тестируемость acceptance criteria |
| `userspec-adequacy-validator` | Реализуемость, over/underengineering |
| `interview-completeness-checker` | Покрытие всех аспектов интервью — нет ли слепых зон |

**Максимум 3 итерации валидации.** После третьей — покажи результаты пользователю, пусть решает.

### Фаза 1.8: Бизнес-анализ [NEW v4.1]

**Цель:** Проработать бизнес-аспекты продукта — юридические требования, монетизацию, стратегию вывода на рынок, аналитику и доступность.

> 💡 **Когда пропустить:** Для Quick Fix и мелких фич. Для Feature (M) — по запросу пользователя. Для Epic/Проект (L) — рекомендуется.

#### Как это работает

1. После User Spec ИИ спрашивает: *«Какие бизнес-блоки нужны для этого проекта?»*
2. Предлагает все 5 блоков с рекомендациями (какие актуальны для конкретного типа проекта)
3. Пользователь выбирает нужные
4. ИИ последовательно прорабатывает каждый выбранный блок
5. Результаты сохраняются в `docs/business-analysis.md`

#### Блок 1.8.1: Legal Compliance (РФ)

Юридические требования к продукту — 152-ФЗ (персональные данные), публичная оферта (437 ГК РФ), 54-ФЗ (онлайн-кассы), 38-ФЗ (реклама), 436-ФЗ (защита детей), 149-ФЗ (UGC).

**Workflow:** AI анализирует тип продукта → задаёт уточняющие вопросы (какие ПД собираются, есть ли платежи, рассылки, UGC, дети) → формирует адаптивный чеклист → даёт рекомендации и подсвечивает особенности законодательства.

**Скилл:** `legal-compliance/SKILL.md`

#### Блок 1.8.2: Monetization

Модель монетизации, ценообразование, выбор платёжной системы, unit economics.

**Workflow:** AI определяет подходящие модели (из 8: SaaS, Freemium, разовая, pay-per-use, реклама, affiliate, донаты, маркетплейс) → задаёт вопросы (как зарабатываем, средний чек, B2B/B2C, география) → формирует тарифную сетку → рассчитывает unit economics (CAC, LTV, LTV/CAC) → рекомендует платёжную систему (ЮKassa / Stripe / CloudPayments) → создаёт задачи на интеграцию.

**Скилл:** `monetization/SKILL.md`

#### Блок 1.8.3: GTM (Go-to-Market)

Стратегия вывода на рынок, основанная на методологии из Obsidian vault.

**Workflow:** AI анализирует продукт → проводит ресёрч конкурентов (веб-поиск + уточнение у пользователя) → определяет тип рынка (красный/синий/зелёный) → выбирает стратегию отстройки → рекомендует каналы привлечения → формирует структуру лендинга → определяет метрики запуска.

**Ключевые принципы:** Конкуренция = доказательство рынка. 80% feedback, 20% продукт. Правило 4 итераций. Продавай до создания продукта.

**Скилл:** `gtm/SKILL.md` + `gtm/references/` (методология)

#### Блок 1.8.4: Analytics

Ключевые метрики, выбор инструмента, проектирование дашборда, алерты.

**Workflow:** AI определяет тип продукта → подбирает метрики по AARRR (Acquisition, Activation, Retention, Revenue, Referral) → рекомендует инструмент (PostHog / Umami / Яндекс.Метрика / Firebase / Grafana) → проектирует дашборд → настраивает алерты (DAU, error rate, conversion, churn, response time) → создаёт задачи на интеграцию.

**Скилл:** `analytics-setup/SKILL.md`

#### Блок 1.8.5: Accessibility

Практичный минимум доступности (не формальное WCAG).

**Workflow:** AI определяет тип интерфейса → если бот/API/CLI — пропуск → если веб/мобилка — формирует чеклист: семантика HTML, контраст ≥4.5:1, клавиатурная навигация, touch-target ≥44×44px, aria-labels.

**Скилл:** `accessibility/SKILL.md`

#### Выходной формат

Результаты всех блоков записываются в `docs/business-analysis.md`. В user-spec и tech-spec добавляются перекрёстные ссылки:

```markdown
> Бизнес-анализ — см. [business-analysis.md](business-analysis.md)
```

### Фаза 2: Техническая спецификация (Tech Spec)

**Цель:** Определить КАК строим — архитектура, стек, зависимости, структура.

#### Подключение субагента: Системный аналитик

На эту фазу ИИ запускает субагент **`system-analyst`** — въедливого и вдумчивого системного аналитика. Этот агент:

- Берёт user-spec как входные данные
- Задаёт **максимально глубокие технические вопросы**
- Анализирует каждое требование на предмет технической реализуемости
- Если пользователь не может ответить → **предлагает варианты** ИЛИ **кладёт в Бэклог**
- Запускает `code-researcher` для сканирования существующего кода (если проект не с нуля)

#### Процесс глубинного технического интервью:

**Цикл 1 — Стек и архитектура:**
- Какой язык/фреймворк предпочитаешь? Есть ли ограничения?
- Монолит или микросервисы? (Для MVP почти всегда монолит)
- Какая база: реляционная (PostgreSQL) или документная (MongoDB)?
- Нужен ли кэш? (Redis)
- Нужна ли очередь задач? (Celery, Bull)
- Какой ORM / query builder?
- REST API или GraphQL? Или и то, и другое?

**Цикл 2 — Модель данных:**
- Какие основные сущности (таблицы)?
- Какие связи между ними? (1:1, 1:N, N:N)
- Какие поля обязательны, какие опциональны?
- Нужен ли soft delete?
- Нужна ли версионность данных (audit log)?
- Какие индексы понадобятся для производительности?

**Цикл 3 — Безопасность:**
- Как реализуем аутентификацию? (JWT, Session, OAuth)
- Какие роли и права доступа?
- Нужно ли шифрование данных at rest?
- Как храним пароли? (bcrypt, argon2)
- Rate limiting нужен?
- CORS-политика?

**Цикл 4 — Инфраструктура и деплой:**
- Docker: multi-stage build?
- Nginx: нужен ли WebSocket proxy?
- SSL: Let's Encrypt или свой сертификат?
- CI/CD: GitHub Actions → какие этапы? (lint, test, build, deploy)
- Environment variables: какие нужны?
- Бэкапы БД: стратегия?
- Health checks: какие эндпоинты?

**Цикл 5 — Масштабирование и производительность:**
- Ожидаемая нагрузка: RPS, количество соединений?
- Нужна ли горизонтальная масштабируемость?
- Кэширование: на каком уровне? (запросы, объекты, страницы)
- CDN для статики?
- Pagination: offset или cursor?
- Connection pooling: настройки?

#### Работа с неотвеченными вопросами:

Точно такой же подход, как в Фазе 1: предложи варианты или положи в Бэклог.

```
Вопрос: Какой стратегии кэширования придерживаемся?

Варианты:
A) Cache-aside (ленивая загрузка) — простая, хорошо для чтения
B) Write-through — консистентная, но дороже
C) TTL-based — простая, 1 час по умолчанию
D) Без кэша пока — добавим когда увидим узкое место

Если не уверен — рекомендую вариант D для MVP.
```

#### Процесс:

### Фаза 4: Реализация

**Цель:** Написать код, убедиться что каждый блок работает.

Есть два режима:

#### Режим A: Последовательная реализация

Для каждой задачи:

1. **Загрузка контекста:**
   ```
   Прочитай task файл → Context Files → Подключи нужные скиллы
   ```

2. **TDD-цикл:**
   ```
   Напиши тесты (из TDD-якоря) → Запусти (красные) →
   Напиши код → Запусти (зелёные) → Рефакторинг
   ```

3. **Ревью (параллельно):**
   - Отправь diff всем ревьюерам из задачи
   - Каждый возвращает JSON-отчёт
   - Исправь findings (максимум 3 раунда)

4. **Документирование:**
   - Запись в `decisions.md` — почему выбрал такое решение
   - Обновление статуса задачи

5. **Коммит:**
   ```bash
   git add [конкретные файлы]
   git commit -m "feat(scope): описание"
   # Если были фиксы по ревью:
   git commit -m "fix(scope): исправления по ревью [round N]"
   ```

#### Режим B: Командная реализация

Для эпиков — когда задач много и они могут идти параллельно:

1. **Team Lead** (Opus) читает tech-spec, строит execution plan
2. Для каждой волны запускает **параллельных Teammates** (Sonnet):
   - Каждый Teammate загружает свой task + контекст
   - Следует TDD-циклу
   - Коммитит код
3. **Ревьюеры** проверяют diff каждого Teammate
4. Teammate исправляет (максимум 3 раунда)
5. Lead фиксирует завершение волны, переходит к следующей

> 💡 **Экономия:** Teammates на Sonnet стоят ~5x дешевле, чем если бы всё делал Opus. Lead на Opus координирует, а рутину делегирует.

> 💡 **Переход к Фазе 5:** После завершения всех волн реализации.

### Фаза 5: Тестирование

**Цель:** Комплексное тестирование проекта по всем необходимым типам.

Это **отдельная выделенная фаза**, а не часть реализации. ИИ самостоятельно определяет, какие типы тестов необходимы для конкретного проекта, и объясняет своё решение.

#### Полный каталог типов тестов

ИИ анализирует проект и принимает решение по каждому типу:

| # | Тип тестов | Описание | Инструменты |
|---|---|---|---|
| 1 | **Smoke** | Базовая проверка: приложение запускается, ключевые эндпоинты отвечают | `pytest`, `curl`, custom scripts |
| 2 | **Unit** | Тестирование отдельных функций и методов в изоляции | `pytest`, `jest`, `unittest` |
| 3 | **Интеграционные** | Проверка взаимодействия между модулями, с БД, с Redis | `pytest` + testcontainers, `supertest` |
| 4 | **Функциональные** | Проверка бизнес-логики: сценарии, user stories, AC | `pytest`, `jest`, BDD-фреймворки |
| 5 | **E2E (End-to-End)** | Полный пользовательский путь от начала до конца | `Playwright`, `Cypress`, `Selenium` |
| 6 | **Snapshot / Visual regression** | Сравнение UI с эталоном — не «поехала» ли вёрстка | `Playwright screenshots`, `Percy`, `Chromatic` |
| 7 | **Security** | Проверка на уязвимости: OWASP Top 10, SQL injection, XSS | `bandit` (Python), `npm audit`, `OWASP ZAP`, `security-auditor` |
| 8 | **Performance** | Время отклика, использование памяти, CPU | `locust`, `k6`, `pytest-benchmark` |
| 9 | **Нагрузочные (Load/Stress)** | Поведение под нагрузкой: сколько RPS выдерживает, где ломается | `locust`, `k6`, `Artillery` |
| 10 | **Contract** | Соответствие API контракту (OpenAPI/Swagger) | `schemathesis`, `dredd`, `pact` |
| 11 | **Accessibility (a11y)** | Доступность: скринридеры, клавиатурная навигация, контрастность | `axe-core`, `pa11y`, `Lighthouse` |
| 12 | **Monkey / Chaos** | Случайные действия пользователя, хаотичные данные | `gremlins.js`, `Chaos Monkey`, fuzz-testing |
| 13 | **Regression** | Старый функционал не сломался после новых изменений | Полный прогон unit + integration + E2E |
| 14 | **Приёмочные (UAT)** | Финальная проверка с точки зрения пользователя перед релизом | Ручное тестирование + `pre-deploy-qa` agent |

#### Как ИИ решает, какие тесты нужны

ИИ анализирует проект по критериям и формирует **Test Plan** с обоснованием:

```markdown
# Test Plan: [Название проекта]

## Обязательные тесты (для ЛЮБОГО проекта):

### ✅ Smoke Tests — ДЕЛАЕМ
Обоснование: Базовая проверка работоспособности нужна всегда.
Покрытие: Запуск приложения, healthcheck, основные эндпоинты.

### ✅ Unit Tests — ДЕЛАЕМ
Обоснование: Бизнес-логика должна быть покрыта тестами.
Покрытие: Сервисы, утилиты, валидация — минимум 80%.

### ✅ Integration Tests — ДЕЛАЕМ
Обоснование: Есть БД и Redis — взаимодействие с ними нужно тестировать.
Покрытие: CRUD-операции, кэширование, транзакции.

### ✅ Security Tests — ДЕЛАЕМ
Обоснование: Есть авторизация и пользовательские данные.
Покрытие: OWASP Top 10, SQL injection, XSS, auth bypass.

### ✅ Regression Tests — ДЕЛАЕМ
Обоснование: Нужна уверенность, что новые фичи не ломают старые.
Покрытие: Полный прогон unit + integration.

## Условные тесты (зависят от проекта):

### ✅ E2E Tests — ДЕЛАЕМ
Обоснование: Есть web-интерфейс → нужны E2E.
Покрытие: Регистрация, логин, основной user flow.

### ✅ Functional Tests — ДЕЛАЕМ
Обоснование: Сложная бизнес-логика → нужно проверить сценарии.
Покрытие: Все user stories из user-spec.

### ⏭️ Snapshot Tests — ПРОПУСКАЕМ
Обоснование: На этапе MVP интерфейс часто меняется,
snapshot-тесты будут постоянно падать. Добавим после стабилизации.
→ [BACKLOG] Добавить snapshot-тесты после стабилизации UI

### ⏭️ Performance Tests — ПРОПУСКАЕМ
Обоснование: Ожидаемая нагрузка < 100 RPS,
преждевременная оптимизация не нужна.
→ [BACKLOG] Добавить performance-тесты при масштабировании

### ⏭️ Load/Stress Tests — ПРОПУСКАЕМ
Обоснование: Аналогично performance — не актуально для MVP.
→ [BACKLOG] Нагрузочное тестирование перед масштабированием

### ⏭️ Contract Tests — ПРОПУСКАЕМ
Обоснование: Один сервис (монолит), нет внешних потребителей API.
→ [BACKLOG] Contract-тесты при появлении внешних клиентов API
```

## 4. Уровень: Эпик — крупная функциональность

Эпик — это набор связанных фич, которые вместе дают пользователю новую возможность. Например: «Система авторизации» = регистрация + логин + восстановление пароля + профиль.

### Процесс работы с эпиком

```
Планирование → Декомпозиция → Wave-параллелизм → Интеграция → Тестирование → QA
```

**Время:** 1-4 часа

#### Шаг 1: Планирование эпика

**Что делать:**
1. Опиши эпик в 2-3 предложениях (что получит пользователь)
2. Запусти `planner.md` + sequential-thinking для архитектуры
3. Определи scope: что входит, что НЕ входит (что НЕ входит → в Бэклог)

**Промпт-пример:**
```
Мне нужна система авторизации для Telegram-бота:
- Регистрация по номеру телефона (через Telegram)
- JWT-токены для API
- Refresh-токены в Redis
- Middleware для проверки авторизации
- Роли: user, admin

Не входит (в бэклог):
- OAuth через внешние провайдеры
- 2FA

Спланируй архитектуру и разбей на задачи.
```

**Инструменты:**
- `agents/planner.md` — планирование
- `sequential-thinking` — пошаговый анализ
- `mgrep` — поиск по существующему коду

#### Шаг 2: Декомпозиция на фичи

Разбей эпик на 3-7 фич, каждая из которых самодостаточна:

```
Эпик: Система авторизации
├── Фича 1: Модели и миграции (User, Session)
├── Фича 2: Сервис авторизации (регистрация, логин)
├── Фича 3: JWT + Refresh токены
├── Фича 4: Middleware авторизации
├── Фича 5: Хендлеры бота (команды /start, /login)
└── Фича 6: Админские команды
```

Сгруппируй в волны по зависимостям:

```
Wave 1: Фича 1 (модели — от них зависят все остальные)
Wave 2: Фича 2 + Фича 3 [параллельно]
Wave 3: Фича 4 + Фича 5 [параллельно]
Wave 4: Фича 6
```

#### Шаг 3: Wave-параллелизм

Для каждой волны:
1. **Запусти задачи параллельно** (субагенты на Sonnet)
2. **Дождись завершения всех задач в волне**
3. **Интеграционная проверка:** работают ли фичи волны вместе
4. **Следующая волна**

#### Шаг 4: Интеграция

- Запусти все тесты: unit + integration
- Проверь, что фичи работают вместе
- Запусти `code-reviewer` на весь diff эпика

#### Шаг 5: Тестирование эпика

ИИ определяет необходимые тесты для эпика (по аналогии с Фазой 5 проекта, но в масштабе эпика):
- Unit-тесты для новых модулей
- Integration-тесты для связей между фичами
- Security-тесты (если эпик связан с авторизацией/данными)
- Regression — убедиться, что старый функционал не сломан

#### Шаг 6: QA эпика

- Все тесты зелёные
- Код-ревью пройден
- UAT по acceptance criteria эпика

## 5. Уровень: Фича — одна конкретная возможность

Фича — основная единица работы. Даже если ты не работаешь на уровне проекта/эпика, каждая нетривиальная задача проходит через этот процесс.

### Процесс работы с фичей

```
Brainstorm → Plan → Approve → Build (блоками) → Test → Review → Commit
```

**Время:** 15-60 минут

**Команда:** `/ship`

#### Шаг 1: Brainstorming

**Что делать:**
- Сформулируй, что именно нужно
- Определи scope и ограничения
- Если есть существующий код — изучи его

**Промпт-пример:**
```
Хочу добавить кэширование ответов API в Redis.

Контекст:
- Бот отвечает на вопросы через OpenAI API
- Одинаковые вопросы часто повторяются
- Нужно кэшировать ответы на 1 час
- Ключ кэша: хэш вопроса

Прочитай текущую реализацию в src/services/ai_service.py
и предложи план.
```

**Инструменты:**
- `mgrep` — найди существующий код по теме
- `context7` — уточни API библиотеки (Redis, etc.)
- Обычный диалог с Claude — проговори подход

#### Шаг 2: Plan Mode

**Что делать:**
1. Claude входит в Plan Mode (если используешь Claude Code)
2. Исследует кодовую базу
3. Предлагает план реализации
4. Ты одобряешь или корректируешь

**Что должен содержать план:**

```markdown
## План: Кэширование ответов в Redis

### Затрагиваемые файлы:
- src/services/ai_service.py (модификация)
- src/services/cache_service.py (новый)
- src/config.py (добавить REDIS_URL)
- tests/test_cache_service.py (новый)

### Шаги:
1. Создать cache_service с get/set/delete
2. Добавить Redis-клиент в config
3. Обернуть вызов OpenAI API в кэш-логику
4. Написать тесты
5. Обновить docker-compose (Redis-сервис)

### Риски:
- Race conditions при параллельных запросах
- Инвалидация кэша при изменении промпта
```

> 💡 **Приём:** Всегда проси Claude показать план ПЕРЕД реализацией. Это в 10 раз дешевле, чем переделывать код.

#### Шаг 3: Approve

Прочитай план. Задай вопросы, если что-то непонятно. Скажи «делай» или скорректируй.

#### Шаг 4: Build (блоками с чекпоинтами)

**Что делать:**
- Работай блоками по 1-3 файла
- После каждого блока — чекпоинт:
  - Файлы сохранены?
  - Линтер проходит?
  - Базовые тесты не сломались?

**Workflow-маркеры** (метки прогресса):

```
[PLAN]      → планирование завершено
[IMPL:1/3]  → реализация блока 1 из 3
[IMPL:2/3]  → реализация блока 2 из 3
[IMPL:3/3]  → реализация блока 3 из 3
[TEST]      → тестирование
[REVIEW]    → код-ревью
[DONE]      → задача завершена
```

**Автоматические проверки при каждом Edit/Write:**
- `security-scan.sh` — сканирование на SQLi, XSS
- `protect-secrets.sh` — проверка на утечку секретов
- `prettier` / `ruff` — автоформатирование

#### Шаг 5: Test

**TDD-подход (рекомендуемый):**

1. Напиши тест на ожидаемое поведение
2. Запусти — должен быть красным
3. Напиши код, чтобы тест стал зелёным
4. Рефакторинг

**Какие тесты для фичи:** ИИ определяет минимально необходимый набор (обычно unit + integration + regression).

#### Шаг 6: Review

**Автоматический код-ревью после >20 строк нового кода:**

Запускается `agents/code-reviewer.md`, который проверяет 10 измерений: корректность, безопасность, производительность, читаемость, тестируемость, DRY, SOLID, error handling, типизация, архитектура.

**Максимум 3 раунда фиксов.** Если после третьего раунда есть критические findings — покажи пользователю.

#### Шаг 7: Commit

```bash
# Стандартный формат (Conventional Commits):
git add src/services/cache_service.py src/config.py tests/test_cache_service.py
git commit -m "feat(cache): добавить кэширование ответов AI в Redis

- Создан cache_service с TTL 1 час
- Redis-клиент в конфиге
- Обёртка вокруг OpenAI API вызова
- Тесты на happy path и edge cases"
```

**Автоматические проверки при коммите:**
- `conventional-commit.sh` — формат сообщения
- `branch-guard.sh` — блокировка push в main
- `security-scan.sh` — финальная проверка

## 6. Уровень: Фикс — быстрое исправление

Фикс — это когда что-то сломалось и надо починить. Или мелкая правка, которая не требует планирования.

### Процесс Quick Fix

```
Определи → Исправь → Проверь
```

**Время:** 5-15 минут

**Команда:** `/bug-fix`

#### Шаг 1: Определи проблему

```
У меня ошибка: [вставь ошибку]
Контекст: [что делал, когда появилась]
```

> 💡 **Приём:** Копируй ПОЛНЫЙ стектрейс. Claude разберётся. Не пересказывай ошибку своими словами — потеряешь детали.

**Инструменты:**
- `mgrep "ErrorMessage"` — найди, где ошибка
- `agents/build-error-resolver.md` — если ошибка сборки

#### Шаг 2: Исправь

- Claude находит проблему
- Предлагает fix
- Применяет (Edit/Write)

**Для сложных багов — используй systematic debugging:**
```
1. Воспроизведи ошибку (запусти тест/команду)
2. Добавь логирование в подозрительные места
3. Сузь область поиска
4. Исправь
5. Удали дебаг-логи
```

#### Шаг 3: Проверь

```bash
# Минимальная проверка:
pytest tests/test_[затронутый_модуль].py

# Плюс regression:
pytest tests/  # весь набор, чтобы не сломать остальное

# Плюс линтер:
ruff check src/
```

#### Шаг 4: Коммит

```bash
git add [файлы]
git commit -m "fix(scope): описание бага и решения"
```

### Когда Quick Fix НЕ подходит:

- Баг требует изменения в 5+ файлах → переходи на уровень **Фича**
- Баг в архитектуре → переходи на уровень **Эпик**
- Непонятно, где баг → systematic debugging, потом решай уровень

## 7. Бэклог проекта

Бэклог — единое место для всех нерешённых вопросов, отложенных задач и идей любого уровня сложности.

### Команда: `/backlog`

| Действие | Команда |
|---|---|
| Показать весь бэклог | `/backlog` |
| Добавить элемент | `/backlog add [описание]` |
| Отметить как выполненный | `/backlog done [ID]` |
| Отфильтровать по типу | `/backlog filter [type]` |
| Приоритизировать | `/backlog prioritize` |

### Структура бэклога

Файл `backlog/BACKLOG.md`:

```markdown
# Бэклог проекта: [Название]

Последнее обновление: 2026-02-24

## 🔴 Высокий приоритет

### BACKLOG-001: Определить стратегию мультиязычности
- **Источник:** User Spec, Фаза 1
- **Тип:** решение
- **Контекст:** Не определились, нужна ли мультиязычность на старте
- **Варианты:** A) Сразу i18n B) Только русский C) Русский + English
- **Статус:** pending
- **Дедлайн:** до релиза v1.1

### BACKLOG-002: Добавить OAuth через Google
- **Источник:** User Spec, Фаза 1
- **Тип:** фича
- **Контекст:** Для MVP хватит JWT, OAuth добавим позже
- **Статус:** pending

## 🟡 Средний приоритет

### BACKLOG-003: Snapshot-тесты после стабилизации UI
- **Источник:** Test Plan, Фаза 5
- **Тип:** тест
- **Контекст:** На этапе MVP интерфейс часто меняется
- **Триггер:** Когда UI стабилизируется (после v1.0)
- **Статус:** pending

### BACKLOG-004: Нагрузочное тестирование
- **Источник:** Test Plan, Фаза 5
- **Тип:** тест
- **Контекст:** Текущая нагрузка < 100 RPS
- **Триггер:** Когда ожидаем > 500 RPS
- **Статус:** pending

## 🟢 Низкий приоритет

### BACKLOG-005: Chaos-тесты при переходе на микросервисы
- **Источник:** Test Plan, Фаза 5
- **Тип:** тест
- **Триггер:** При переходе от монолита к микросервисам
- **Статус:** pending

## ✅ Выполнено

### BACKLOG-000: Пример выполненного элемента
- **Выполнено:** 2026-02-24
- **Результат:** Реализовано в task-007
```

### Типы элементов бэклога

| Тип | Описание | Пример |
|---|---|---|
| `решение` | Нерешённый вопрос, требующий выбора | Какую БД выбрать для аналитики |
| `фича` | Отложенная функциональность | OAuth через Google |
| `тест` | Отложенный тип тестирования | Нагрузочные тесты |
| `баг` | Известный баг, который не критичен | Кнопка чуть съезжает на мобилке |
| `улучшение` | Рефакторинг, оптимизация | Переписать кэш на Redis Streams |
| `идея` | Непроработанная идея | Может, добавить Telegram-бота? |
| `долг` | Технический долг | Заменить raw SQL на ORM |

### Когда элементы попадают в бэклог

1. **Фаза 1 (User Spec):** Вопросы, на которые пользователь не ответил
2. **Фаза 2 (Tech Spec):** Технические решения, отложенные до масштабирования
3. **Фаза 5 (Тестирование):** Типы тестов, которые не нужны сейчас, но понадобятся позже
4. **Любой момент:** Идеи, замечания, желания, которые не входят в текущий scope

### Приоритизация бэклога

Команда `/backlog prioritize` запускает ИИ, который:
1. Анализирует все элементы бэклога
2. Проверяет, не наступил ли триггер для отложенных элементов
3. Предлагает переприоритизацию
4. Рекомендует, что взять в следующий спринт/итерацию

### Матрица: ЧТО → ЧЕМ → КОГДА

| Что проверяем | Инструмент | Когда | Авто? |
|---|---|---|---|
| Безопасность кода | `security-scan.sh` + `security-auditor` | Каждый Write/Edit | ✅ |
| Защита секретов | `protect-secrets.sh` | Каждый Write/Edit | ✅ |
| Деструктивные операции | `destructive-guard.sh` | Каждый Bash | ✅ |
| Форматирование JS/TS | `prettier` | Каждый Write/Edit .js/.ts | ✅ |
| Форматирование Python | `python-lint.sh` (ruff) | Каждый Write/Edit .py | ✅ |

| Типы Python | `pyright-lsp` | Постоянно | ✅ |
| Упрощение кода | `code-simplifier` | После 3+ изменённых файлов | ✅ |
| Код-ревью | `code-reviewer` | После кода >20 строк | ❌ (по вызову) |
| Anti-mirage | `mgrep` / `reality-checker` | Перед ссылкой на файл/функцию/API | ❌ (по правилу) |
| Тесты | `tdd-guide` | Feature / Epic | ❌ (по вызову) |
| Архитектура | `planner` + sequential-thinking | Feature / Epic | ❌ (по вызову) |
| User-spec | 3 валидатора параллельно | Фаза 1 (Проект) | ❌ (по вызову) |
| Tech-spec | 5 валидаторов параллельно | Фаза 2 (Проект) | ❌ (по вызову) |
| Задачи | 2 валидатора параллельно | Фаза 3 (Проект) | ❌ (по вызову) |

### Правило трёх попыток

На каждом уровне валидации максимум 3 итерации:

```
Итерация 1: Валидация → findings → fix
Итерация 2: Ре-валидация → findings → fix
Итерация 3: Финальная валидация → если ещё есть issues → покажи лог пользователю
```

Принцип: **лучше ложно-положительное, чем ложно-отрицательное.** Проверенная ошибка дешевле пропущенной.

## 11. Anti-Mirage — проверка реальности

Claude любит «галлюцинировать» — ссылаться на несуществующие файлы, функции, API и пакеты. Anti-Mirage — набор правил, которые это предотвращают.

### Правила

| Перед чем | Что проверить | Как |
|---|---|---|
| Ссылкой на файл | Файл существует | `mgrep` или `ls` / `find` |
| Вызовом функции | Функция существует в кодовой базе | `mgrep "functionName"` |
| Импортом пакета | Пакет в зависимостях | `package.json` / `pyproject.toml` |
| Ссылкой на поле БД | Поле есть в схеме | schema / migrations |
| Использованием API | Эндпоинт существует и работает | Документация / `context7` |
| Упоминанием конфига | Переменная определена | `.env.example` / config файлы |

### Как интегрировать

В `CLAUDE.md` или `rules/anti-mirage.md`:

```markdown
## Anti-Mirage Rules

ПЕРЕД ссылкой на любой файл, функцию, API или пакет — ПРОВЕРЬ:
1. Файл: mgrep или ls/find
2. Функция: mgrep "functionName" в кодовой базе
3. Пакет: проверь package.json / pyproject.toml
4. Поле БД: проверь schema / migrations
5. API: проверь документацию через context7

Если НЕ УВЕРЕН — честно скажи: «Не знаю, надо проверить»
НИКОГДА не выдумывай имена файлов или функций.
```

### Типичные миражи Claude

| Тип | Пример | Как ловить |
|---|---|---|
| Несуществующий файл | «Отредактируй `src/utils/helpers.py`» (его нет) | `ls src/utils/` |
| Несуществующая функция | «Вызови `user.get_full_name()`» (метода нет) | `mgrep "get_full_name"` |
| Устаревший API | «Используй `requests.get_async()`» (нет такого) | `context7` — актуальная документация |
| Несуществующий пакет | «Установи `fastapi-cache3`» (не существует) | `pip search` / PyPI |
| Фантомное поле БД | «Добавь `WHERE user.role = 'admin'`» (поля role нет) | Проверь миграции |

## 12. Auto-Fix Pipeline

Автоматическое исправление ошибок в 4 уровня. Максимум 3 попытки на каждом уровне. После — покажи лог пользователю.

### Level 1: Commit упал

```
Ошибка → Прочитай сообщение → Исправь → Коммить заново
```

Типичные причины: линтер ругается, формат коммит-сообщения неверный, security scan нашёл проблему.

### Level 2: Push отклонён

```
Ошибка → Rebase на свежую main → Push заново
       → Если конфликты — создай PR
```

Типичные причины: remote изменился, branch protection rules.

### Level 3: CI/CD упал

```
gh run view --log-failed → Прочитай ошибку → Fix-коммит → Push
```

Типичные причины: тесты падают в CI, отсутствуют env vars, Docker не собирается.

### Level 4: Деплой упал

```
Логи контейнера → Определи причину → Fix → Рестарт
(максимум 3 попытки → если не помогло — revert)
```

Типичные причины: миграция БД упала, порт занят, out of memory.

### Схема пайплайна

```
Commit упал? ──yes──→ Прочитай ошибку → Fix → Повтори (x3)
     │                                              │
     no                                        Не помогло?
     ↓                                              ↓
Push отклонён? ─yes─→ Rebase/PR → Push (x3)    Покажи лог
     │                                          пользователю
     no
     ↓
CI упал? ──yes──→ gh run view → Fix-коммит (x3)
     │
     no
     ↓
Deploy упал? ─yes─→ Логи → Fix → Рестарт (x3) → Revert
     │
     no
     ↓
  ✅ Успех!
```

## 13. Паттерны и приёмы вайбкодера

### Паттерн 1: Двунаправленная трассировка требований

Проверяй в обе стороны:
- **Forward:** Каждое требование из user-spec отражено в tech-spec? Ничего не потеряли?
- **Reverse:** Каждая задача в tech-spec обоснована требованием? Нет ли лишнего?

### Паттерн 2: TDD-якоря в спецификациях

Не жди фазы реализации — опиши тесты уже на уровне tech-spec:

```markdown
## Task: Регистрация пользователя

### TDD-якорь:
- test_register_creates_user — при валидных данных создаётся пользователь
- test_register_duplicate_email — при повторном email возвращает 409
- test_register_weak_password — при слабом пароле возвращает 400
```

### Паттерн 3: Context Files (Just-in-Time загрузка)

Не грузи всё — грузи только то, что нужно для конкретной задачи. Это экономит контекстное окно и улучшает фокус модели.

### Паттерн 4: Decisions.md — лог решений

Записывай, ПОЧЕМУ принял то или иное решение:

```markdown
## 2026-02-24: Выбор Redis для кэширования

**Контекст:** Нужен кэш для ответов AI API
**Альтернативы:** файловый кэш, Memcached, Redis
**Решение:** Redis — уже в стеке, TTL, pub/sub на будущее
**Последствия:** +Redis в docker-compose
```

### Паттерн 5: Wave-параллелизм

Группируй задачи в волны по зависимостям. Субагенты на Sonnet работают параллельно, Lead на Opus координирует.

### Паттерн 6: Максимум 3 итерации

Жёсткий лимит на любой цикл: валидация, ревью, auto-fix. После трёх — эскалация к человеку.

### Паттерн 7: Контекст после Compaction

Hook `reinject-context.sh` автоматически подгружает текущую задачу, активные файлы и прогресс после compaction.

### Паттерн 8: Промпт = конкретика

Плохо: «Сделай авторизацию»
Хорошо: конкретный стек, файлы, формат токенов, модели, пути.

### Паттерн 9: Cost-Awareness

Sonnet для рутины, Opus для сложного. Не грузи лишний контекст. `cost-tracker.sh` считает стоимость.

### Паттерн 10: Atomic Commits

Один коммит = одно логическое изменение. Conventional Commits: `feat`, `fix`, `test`, `refactor`, `docs`.

### Паттерн 11: Бэклог вместо «потом сделаю»

Всё, что «потом» — в бэклог. Нерешённый вопрос — в бэклог. Отложенный тест — в бэклог. Идея на будущее — в бэклог. Ничего не теряется.

### Паттерн 12: Глубинное интервью > Поверхностный бриф

Субагенты бизнес-аналитик и системный аналитик задают десятки вопросов, а не 3-4. Это дорого по токенам, но на порядок дешевле, чем переделывать проект.

## 15. Security Pipeline [NEW v4.1]

Автоматическое сканирование зависимостей и конфигурации на уязвимости. Два инструмента работают в связке.

### Инструменты

| Инструмент | Что делает | Язык |
|---|---|---|
| **vet** | Сканирует зависимости на уязвимости (CVE), malware, typosquatting | Python, Node.js, Go |
| **agents-lint** | Проверяет конфигурацию LLM-агентов на prompt injection, чрезмерные permissions | YAML, JSON, TOML |

### Когда запускать

| Триггер | Действие |
|---|---|
| Добавление зависимости (`pip install`, `npm install`) | `vet` |
| Обновление зависимостей | `vet` |
| Создание/изменение конфига агента | `agents-lint` |
| Перед деплоем (Gate 3) | Оба |
| По запросу `/security-check` | Оба |

### CI/CD интеграция

Security Pipeline интегрирован в GitHub Actions:

```yaml
# .github/workflows/security.yml
name: Security Scan
on: [push, pull_request]
jobs:
  security:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: vet scan
        run: vet scan --format=json
      - name: agents-lint
        run: agents-lint check .
```

### Quality Gates с Security Pipeline

| Gate | Без Security Pipeline | С Security Pipeline (v4.1) |
|---|---|---|
| Gate 1 (Plan) | Архитектура + security review | + agents-lint для конфигов агентов |
| Gate 2 (Code) | Code review + anti-mirage | + vet для новых зависимостей |
| Gate 3 (Deploy) | Чеклист деплоя | + полный security scan (vet + agents-lint) |
| Gate 4 (Skill) | Skill quality gate | + проверка безопасности скилла |
| Gate 5 (Business) | — | Проверка полноты бизнес-анализа |
