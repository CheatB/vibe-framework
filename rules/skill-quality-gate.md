# Обязательный Quality Gate для скиллов

**ТРИГГЕР:** Любое создание нового скилла или значительное изменение существующего SKILL.md.

Это правило работает **поверх** skill-creator — добавляет обязательный этап проверки.

---

## Когда срабатывает

- После команды "создай скилл" / "сделай навык" / skill-creator
- После ручного создания SKILL.md
- После значительного редактирования SKILL.md (>20% контента)
- НЕ срабатывает при мелких правках (опечатки, форматирование)

## Формула качества (из мега-отчёта оценки 68 скиллов)

```
Score ≈ 70 + (примеры × 10) + (framework × 5) + (self-contained × 5) + (workflow × 5)
```

Каждый фактор = 0 или 1:
- **примеры** — есть ли минимум 2 конкретных примера ввода/вывода в SKILL.md
- **framework** — используется ли структурированный подход (шаблоны, rubric, пошаговый процесс)
- **self-contained** — работает ли скилл без внешних зависимостей
- **workflow** — есть ли чёткий пошаговый workflow для пользователя

**Минимальный проходной балл: 85%**

## Процедура проверки (5 шагов)

### Шаг 1: Структурный чеклист

Проверь SKILL.md на наличие:

```
[ ] Frontmatter с name и description (description «пушит» использование)
[ ] Минимум 2 конкретных примера (ввод → вывод или шаблон)
[ ] Секция workflow (пошаговый процесс)
[ ] Секция "Приоритет при конфликтах" (как вести себя при противоречиях)
[ ] Нет зависимости от недоступных ресурсов
[ ] Размер SKILL.md < 500 строк (если больше — ссылки на references/)
```

Если >2 пунктов не прошли → **исправь перед продолжением**.

### Шаг 2: Sub-agent ревью (параллельно)

Запусти 3 проверяющих через Task tool параллельно:

**[code-reviewer]** — качество инструкций:
- Инструкции конкретные или абстрактные?
- Примеры рабочие или "рыба"?
- Есть ли противоречия в тексте?

**[architect]** — архитектура скилла:
- Progressive disclosure (metadata → SKILL.md → references)?
- Правильная структура директорий?
- Масштабируется ли подход?

**[security-reviewer]** — безопасность:
- Нет хардкоженных секретов?
- Нет eval()/exec() без санитизации?
- Нет инъекций через пользовательский ввод?

### Шаг 3: Тест-прогон

Если скилл можно протестировать автоматически:

1. Сгенерируй 2-3 реалистичных промпта для скилла
2. Мысленно прогони скилл через каждый промпт
3. Проверь: даёт ли SKILL.md достаточно информации для качественного ответа?
4. Если нет — допиши недостающее

Если скилл субъективный (стиль, дизайн) — покажи пользователю пример вывода.

### Шаг 4: Расчёт score

Посчитай итоговый балл:

```
base = 70
+ 10 если есть ≥2 конкретных примера
+  5 если есть framework/структура
+  5 если self-contained
+  5 если есть чёткий workflow
+  5 бонус за "Приоритет при конфликтах"
= итого (макс 100)
```

### Шаг 5: Решение

```
Score ≥ 85%  → ✅ PASS — скилл готов к использованию
Score 75-84% → ⚠️ WARN — показать пользователю, предложить доработку
Score < 75%  → ❌ FAIL — обязательная доработка перед использованием
```

## Формат вывода

После проверки показать пользователю:

```
━━━ 🎯 SKILL QUALITY GATE ━━━

Скилл: [название]
Score: [N]% [✅ PASS / ⚠️ WARN / ❌ FAIL]

Чеклист:
✅ Frontmatter
✅ Примеры (3 шт.)
✅ Workflow
⚠️ Нет секции "Приоритет при конфликтах"
✅ Self-contained

Ревью:
[code-reviewer]: ✅ Инструкции конкретные
[architect]: ✅ Структура корректная
[security-reviewer]: ✅ Безопасно

Рекомендации:
- Добавить секцию "Приоритет при конфликтах"
```

## Интеграция с skill-creator

Если скилл создаётся через skill-creator:
1. Quality Gate запускается **после** шага "Package and Present"
2. Если FAIL → вернуться в цикл улучшения
3. Если PASS → скилл готов

Если скилл создаётся вручную:
1. Quality Gate запускается при первом использовании скилла
2. Или по запросу пользователя

## Исключения

Пользователь может пропустить проверку командой:
```
"пропусти quality gate" / "skip eval"
```

Но Claude покажет предупреждение:
```
⚠️ Quality Gate пропущен по запросу.
Скилл может работать нестабильно.
```

## Паттерны из мега-оценки

Что делает скилл отличным (90%+):
1. Конкретные примеры в SKILL.md (шаблоны, код, output samples)
2. Чёткий workflow — пошаговый процесс, не абстрактные рекомендации
3. Framework-based — RICE, MoSCoW, SMART, rubric scoring
4. Self-contained — работает без внешних зависимостей

Что снижает качество (<85%):
1. Зависимость от внешних источников (CONNECTORS.md, MCP серверы)
2. Отсутствие примеров вывода
3. Абстрактные инструкции вместо конкретных шагов
4. Требование специфического окружения
