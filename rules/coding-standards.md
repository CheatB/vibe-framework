# Стандарты кодирования

## Правило №0: Прочитай перед кодингом

Перед любым изменением кода — ОБЯЗАТЕЛЬНО:

1. **Прочитай CLAUDE.md проекта** (если есть) — архитектура, паттерны, решения
2. **Grep usages** изменяемых функций/классов — кто их использует?
3. **Прочитай файлы-зависимости** — что импортирует этот файл, кто его импортирует
4. **Проверь существующие тесты** — запусти тесты для изменяемой области
5. **Ищи переиспользуемый код** — не пиши с нуля если похожее уже есть

Если находишь проблемы или конфликты → обсуди с пользователем ДО начала кодинга.

## Язык кода
- Весь код на **русском языке**:
  - Комментарии на русском
  - Названия переменных на русском (где уместно)
  - Docstrings на русском
  - Сообщения пользователю на русском

## Стиль кода

### Общие правила
- **DRY** (Don't Repeat Yourself) — не дублируй код
- **KISS** (Keep It Simple, Stupid) — делай просто
- **YAGNI** (You Aren't Gonna Need It) — не пиши код "на будущее"

### Размеры файлов
- ❌ Файл > 500 строк — разбить на модули
- ❌ Функция > 50 строк — разбить на части
- ✅ Один файл = одна ответственность

### Именование
```typescript
// ✅ Хорошо
const пользовательИмя = "Иван";
function получитьДанныеПользователя() {}

// ❌ Плохо
const a = "Иван";
function getData() {}
const temp = 123;
```

### Структура проекта
```
app/
├── handlers/     # Обработчики
├── services/     # Бизнес-логика
├── database/     # Работа с БД
├── utils/        # Утилиты
├── types/        # TypeScript типы
└── tests/        # Тесты
```

## Безопасность

### ЗАПРЕЩЕНО
- ❌ Хардкодить секреты в коде
- ❌ Коммитить `.env` файлы
- ❌ Использовать `eval()` или подобное
- ❌ Хранить пароли в открытом виде

### ОБЯЗАТЕЛЬНО
- ✅ Использовать `.env` для секретов
- ✅ Валидировать пользовательский ввод
- ✅ Использовать prepared statements для SQL
- ✅ Проверять права доступа

## Тестирование

### Покрытие
- Минимум **80%** покрытия тестами
- Тесты для всех публичных функций
- Edge cases покрыты

### TDD подход
1. Сначала пиши тест
2. Тест падает (RED)
3. Пиши код
4. Тест проходит (GREEN)
5. Рефакторинг

## Git workflow

### Коммиты
Формат: `<тип>: <описание>`

Типы:
- `feat:` — новая функция
- `fix:` — исправление бага
- `refactor:` — рефакторинг
- `docs:` — документация
- `test:` — тесты
- `chore:` — рутина (зависимости и т.д.)

Примеры:
```
feat: добавить регистрацию пользователя
fix: исправить ошибку валидации email
refactor: упростить функцию расчёта скидки
```

### Что НЕ коммитить
- ❌ `console.log`, `debugger`, `print()` для отладки
- ❌ Закомментированный код
- ❌ Временные файлы
- ❌ `node_modules`, `.env`

## TypeScript

### Типизация
```typescript
// ✅ Хорошо
interface Пользователь {
  имя: string;
  возраст: number;
}

function создатьПользователя(данные: Пользователь): Пользователь {
  return данные;
}

// ❌ Плохо
function createUser(data: any): any {
  return data;
}
```

### Избегать
- ❌ `any` (используй `unknown` если тип неизвестен)
- ❌ Type assertions без необходимости
- ❌ Игнорирование ошибок типов

## Производительность

### Оптимизация
- ✅ Используй `useMemo` для тяжёлых вычислений
- ✅ Используй `useCallback` для функций в зависимостях
- ✅ Оптимизируй запросы к БД (индексы, JOIN)
- ✅ Lazy loading для больших компонентов

### Избегать
- ❌ Лишние ререндеры в React
- ❌ N+1 запросы к БД
- ❌ Загрузка всех данных сразу (используй пагинацию)
