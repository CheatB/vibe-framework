# /docs — Управление документацией проекта

Команда для аудита, обновления и проверки качества проектной документации.

## Использование

```
/docs              — показать статус документации
/docs audit        — полный аудит качества
/docs update       — обновить после изменений
/docs check        — проверить консистентность
```

## /docs (без аргументов) — Статус

1. Проверь наличие файлов в project-knowledge:
   - project.md
   - architecture.md
   - patterns.md
   - deployment.md
   - ux-guidelines.md (опционально)
2. Для каждого файла определи состояние:
   - ✅ Заполнен — реальный контент, актуален
   - ⚠️ Частично заполнен — есть плейсхолдеры или пустые секции
   - 📝 Шаблон — только структура без контента
   - ❌ Отсутствует — файла нет
3. Покажи сводку:
   ```
   📚 Документация проекта: [название]

   project.md       ✅ Заполнен (2.1 KB)
   architecture.md  ⚠️ Частично (3.4 KB, 2 плейсхолдера)
   patterns.md      ✅ Заполнен (1.8 KB)
   deployment.md    📝 Шаблон
   ux-guidelines.md ❌ Отсутствует

   Рекомендации:
   - Заполнить deployment.md перед деплоем
   - Убрать плейсхолдеры в architecture.md
   ```

## /docs audit — Полный аудит

1. Прочитай ВСЕ файлы: project-knowledge + CLAUDE.md + README.md
2. Проверь по чеклисту (см. rules/documentation.md):
   - Блоки кода в документации → заменить ссылками
   - Generic знания фреймворка → удалить
   - Детали уровня функций → перенести в код
   - Раздутые секции (>5KB) → сжать
   - Дублирование между файлами → консолидировать
   - Плейсхолдеры/TODO → заполнить или удалить
   - Устаревшая информация → обновить
   - Универсальные паттерны в patterns.md → убрать
3. Сохранить операционные детали
4. Показать отчёт:
   ```
   🔍 Аудит документации

   Найдено проблем: [N]

   architecture.md:
   ⚠️ Блок кода на строке 45 → заменить ссылкой на файл
   ⚠️ Секция "Data Model" > 80 строк → вынести в database.md

   patterns.md:
   ❌ Универсальный паттерн "DRY" → удалить (есть в coding-standards.md)

   Исправить? [да/нет/выборочно]
   ```
5. Применить одобренные исправления

## /docs update — Обновление после изменений

1. Определи что изменилось:
   - Новые файлы/модули → обновить architecture.md (структура)
   - Новые зависимости → обновить architecture.md (зависимости)
   - Новые env vars → обновить deployment.md
   - Новые паттерны → обновить patterns.md
   - Изменения стека → обновить architecture.md
2. Обнови затронутые файлы
3. Проверь перекрёстные ссылки
4. Покажи diff изменений

## /docs check — Консистентность

1. Прочитай все файлы project-knowledge
2. Извлеки: названия технологий/версий, сервисов, БД, env vars, платформ
3. Найди несоответствия:
   - "PostgreSQL" vs "Postgres" vs "postgres"
   - Разные версии одной библиотеки
   - Разные названия одного сервиса
4. Покажи отчёт → спроси правильный вариант → стандартизируй

## Принципы

- Документация ≠ код. Ссылки вместо копирования кода.
- CLAUDE.md и README.md — указатели, не контейнеры.
- Один факт — одно место. Дублирование = несоответствие.
- Файл > 5KB — подозрительно. Вероятно нужно разбить.
