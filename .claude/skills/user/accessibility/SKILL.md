# Accessibility (a11y) — Практичный минимум

Скилл для обеспечения базовой доступности веб-приложений и мобильных интерфейсов.

## Когда использовать

- Фаза 1.8.5 в `/new-project` pipeline
- Standalone через `/business-analysis`
- При создании/ревью UI-компонентов

## Когда НЕ использовать

- Telegram-бот → пропуск (a11y не применимо)
- API-сервис → пропуск
- CLI-утилита → пропуск
- Только бэкенд → пропуск

## Workflow

```
1. Определить тип интерфейса
2. Если бот/API/CLI → пропуск с пометкой "a11y не применимо"
3. Если веб/мобилка → сформировать чеклист
4. Создать задачи на проверку
5. Записать результат в docs/business-analysis.md → секция "Accessibility"
```

## Чеклист (практичный минимум)

### 1. Семантика HTML

- [ ] Правильная иерархия заголовков (h1→h2→h3, не пропускать уровни)
- [ ] Семантические теги (`nav`, `main`, `aside`, `footer`, `article`, `section`)
- [ ] `alt`-тексты для всех информативных изображений
- [ ] `label` для всех полей форм (не placeholder вместо label!)
- [ ] Кнопки — `<button>`, ссылки — `<a>`, не `<div onclick>`

### 2. Контраст и читаемость

- [ ] Контраст текста к фону ≥ **4.5:1** (обычный текст)
- [ ] Контраст ≥ **3:1** (крупный текст ≥18px bold или ≥24px)
- [ ] Размер шрифта ≥ **16px** для основного текста
- [ ] Не использовать цвет как **единственный** индикатор (добавить иконку/текст)
- [ ] Достаточный line-height (≥1.5 для основного текста)

### 3. Клавиатурная навигация

- [ ] Все интерактивные элементы доступны с клавиатуры (Tab)
- [ ] Видимый **focus-индикатор** (не `outline: none` без замены!)
- [ ] Логичный порядок табуляции (не прыгает хаотично)
- [ ] **Escape** закрывает модальные окна
- [ ] **Enter/Space** активирует кнопки и ссылки
- [ ] Skip-link для навигации (опционально, но полезно)

### 4. Мобильная доступность

- [ ] Touch-target ≥ **44×44px** (минимум для пальца)
- [ ] Адаптивная вёрстка (без горизонтального скролла на 375px+)
- [ ] Читаемость без зума
- [ ] Правильные `input type` для мобильных (`email`, `tel`, `number`)
- [ ] Нет hover-only интеракций (на мобильных нет ховера)

### 5. Скринридер

- [ ] `aria-label` для кнопок-иконок (без текста)
- [ ] `role` для кастомных виджетов (табы, слайдеры, дропдауны)
- [ ] `aria-live` для динамического контента (уведомления, чат)
- [ ] `aria-hidden="true"` для декоративных элементов
- [ ] Язык страницы указан (`<html lang="ru">`)

## Инструменты проверки

| Инструмент | Что проверяет | Как запустить |
|---|---|---|
| **Chrome Lighthouse** | Комплексный a11y аудит | DevTools → Lighthouse → Accessibility |
| **axe DevTools** | Детальная проверка WCAG | Расширение Chrome |
| **WAVE** | Визуальная проверка | wave.webaim.org |
| **Contrast Checker** | Контраст цветов | webaim.org/resources/contrastchecker |
| **Keyboard Navigation** | Ручная проверка Tab/Enter/Escape | Просто убери мышь и навигируй |

## Пример вывода

```markdown
## Accessibility

### Тип интерфейса: Веб-приложение (React)

### Применимые проверки
✅ Семантика HTML — 5 пунктов
✅ Контраст — 5 пунктов
✅ Клавиатура — 6 пунктов
✅ Мобильная — 5 пунктов
✅ Скринридер — 5 пунктов

### Задачи (→ tech-spec / tasks.md)
- [ ] Проверить контраст (Lighthouse → score ≥90)
- [ ] Добавить aria-labels к кнопкам-иконкам
- [ ] Протестировать клавиатурную навигацию
- [ ] Проверить на мобильных (минимум 375px)
- [ ] Добавить skip-link на главную навигацию
```

## Пример: Бот → пропуск

```markdown
## Accessibility

### Тип интерфейса: Telegram-бот
### Статус: ⏭️ Пропущено — a11y не применимо к ботам

Telegram сам обеспечивает доступность своего интерфейса.
Если в будущем появится веб-интерфейс → вернуться к этому блоку.
```

## Приоритет при конфликтах

1. Контраст и читаемость > красивый дизайн (если конфликтуют)
2. Семантический HTML > кастомные компоненты (проще и доступнее)
3. Практичность > формальное соответствие WCAG (не нужна сертификация, нужно удобство)
4. Если веб-интерфейса нет → честно пропустить блок, не создавать ненужные задачи
