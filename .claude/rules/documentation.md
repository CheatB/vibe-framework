# Правила документации

## Философия

**Цель:** открыл документацию → понял проект без чтения кода.

Что за проект, как устроен, что делает, где что лежит, как деплоить, где логи.
Навигационный гид высокого уровня.

## Что документировать

### ОБЯЗАТЕЛЬНО
- Что существует, что делает, и ПОЧЕМУ так сделано
- Обзор компонентов и как они связаны
- Принятые решения (почему этот стек, эта архитектура)
- Операционные детали (адреса серверов, процедуры деплоя, пути логов, env vars)
- Ограничения scope (что проект НЕ делает)

### НЕ документировать
- Сигнатуры функций (видно из кода)
- Детали реализации (видно из кода)
- Стандартное поведение фреймворка (есть в офф. документации)
- Код в документации (устаревает и раздувает контекст)

## Формат

### Без кода в документации
```markdown
<!-- ❌ Плохо -->
## Авторизация
```python
def verify_token(token):
    ...
```

<!-- ✅ Хорошо -->
## Авторизация
JWT-токены, проверка в [auth.ts:45-67](src/auth/jwt.ts#L45-L67).
Refresh token через httpOnly cookie, время жизни 7 дней.
```

### Без дублирования
Информация живёт в ОДНОМ месте. Перекрёстные ссылки:
> "Переменные окружения — см. deployment.md"

### Только проектно-специфичное
`patterns.md` содержит ТОЛЬКО уникальные паттерны ЭТОГО проекта.
Универсальные стандарты кодирования — в `rules/coding-standards.md`.

## Структура Project Knowledge

Каждый проект содержит 4 обязательных + опциональные файлы:

| Файл | Содержание |
|------|-----------|
| project.md | Обзор, аудитория, проблема, 3-5 ключевых фич, scope |
| architecture.md | Стек (+ ПОЧЕМУ), структура, зависимости, интеграции, модель данных |
| patterns.md | Проектные паттерны, git workflow, методы тестирования, бизнес-правила |
| deployment.md | Платформа, env vars, CI/CD триггеры, rollback, мониторинг |

Опционально:
- **ux-guidelines.md** — только для проектов со значительным UI
- **{domain}.md** — домен-специфичные (bot.md, vault.md, api.md)

## Размеры и масштабирование

- Каждый файл < 5KB (подозрительно если больше)
- Секция > 80 строк → выносить в отдельный файл
- Ссылки вместо дублирования
- Если проект растёт → выносить подсекции: `database.md`, `monitoring.md`, `git-workflow.md`

## CLAUDE.md и README.md

### CLAUDE.md (для AI-агентов)
Минимальная точка входа:
- Название проекта, одна строка описания
- Ссылка на project-knowledge
- Путь к бэклогу
- Дефолтная ветка
- Ключевые правила (5-7 строк максимум)

### README.md (для людей, на русском)
- Название, назначение
- Обзор структуры папок
- Как запустить локально
- Ссылка на документацию

**Оба файла — указатели, НЕ контейнеры информации.**

## Workflow: аудит документации

1. Прочитай все файлы из project-knowledge + CLAUDE.md + README.md
2. Найди проблемы:
   - Блоки кода → заменить ссылками на файлы
   - Общие знания фреймворка → удалить
   - Детали уровня функций → перенести в комментарии кода
   - Раздутые секции (>5KB) → сжать
   - Дублирование → консолидировать
   - Плейсхолдеры (`TODO`, `[название]`) → заполнить или удалить
   - Устаревшая инфо (файлы/функции которых нет) → обновить
3. Сохранить операционные детали (адреса серверов, SSH конфиги, пути логов)
4. Сформировать отчёт → спросить пользователя → применить

## Обновление документации

### Когда обновлять
- После завершения фичи (через /done)
- После изменения архитектуры
- После изменения стека/зависимостей
- После изменения процесса деплоя
- После принятия важного решения (→ DECISIONS.md)

### DECISIONS.md
Журнал стратегических решений:
```markdown
## [YYYY-MM-DD] Название решения

**Контекст:** Что привело к решению
**Решение:** Что решили
**Альтернативы:** Что рассматривали
**Последствия:** Что это меняет
```

## Чеклист качества документации

- [ ] Можно понять проект не читая код
- [ ] Нет блоков кода (только ссылки на файлы)
- [ ] Нет дублирования между файлами
- [ ] Нет generic знаний фреймворка
- [ ] Все env vars описаны (имена, назначение, НЕ значения)
- [ ] Операционные детали актуальны
- [ ] Каждый файл < 5KB
