# Правила тестирования

## Пирамида тестов

```
        /\
       /E2E\        ← Мало (3-5 критичных сценариев)
      /------\
     /Интегр.\      ← Средне (все эндпоинты + БД)
    /----------\
   /   Unit     \   ← Много (вся бизнес-логика)
  /--------------\
 /    Smoke      \  ← Минимум (1-2 базовых теста)
/------------------\
```

## Когда какие тесты писать

### Smoke-тесты
- Фреймворк настроен и работает
- Переменные окружения доступны
- Базовые импорты работают
- **Когда:** один раз при настройке проекта

### Unit-тесты
- Функции с расчётами, валидацией, трансформацией
- Логика принятия решений (if/else, switch)
- Обработка ошибок
- **Пропустить:** простые геттеры/сеттеры, однострочные изменения

### Интеграционные тесты
- Все API-эндпоинты (особенно POST/PUT/DELETE)
- Операции с БД (создание/обновление/удаление)
- Интеграции с внешними сервисами (платежи, email, webhook)
- **Правило:** каждый эндпоинт и каждая запись в БД = интеграционный тест

### E2E-тесты
- Топ 3-5 критичных пользовательских сценариев
- Большие фичи (>5 задач)
- Критичные бизнес-процессы (авторизация, платежи)
- **Когда:** после деплоя на dev, перед ручным тестированием

## Decision Framework

### Нужны ли unit-тесты?
**ДА если:** функция с бизнес-логикой, принимает решения, трансформирует данные, обрабатывает ошибки.
**НЕТ если:** простой getter/setter, однострочное изменение, тривиальный конфиг.

### Нужны ли интеграционные тесты?
**ДА если:** tech-spec требует, есть API-эндпоинты, работа с БД, вызов внешних сервисов.
**НЕТ если:** tech-spec говорит "None", чисто клиентская фича.

### Нужны ли E2E-тесты?
**ДА если:** >5 задач, критичный flow, breaking changes, пользователь просит.
**НЕТ если:** маленькая фича (<3 задач), некритичный функционал.

## Ключевые принципы

1. **Тесты сразу** — в той же сессии, до перехода к следующей задаче
2. **Тестируй поведение, не реализацию** — фокус на ЧТО, не КАК
3. **Быстрые тесты** — unit: миллисекунды, integration: секунды, E2E: минуты
4. **Изоляция** — мокай внешние зависимости в unit-тестах
5. **Один тест = одна проверка** — каждый тест валидирует одну вещь
6. **Понятные имена** — описывай что тестируется и ожидаемый результат
7. **Чистое состояние** — каждый тест стартует с известным состоянием БД
8. **Каждый тест оправдывает существование** — ловит ошибку, которую не ловят другие

## Антипаттерны

### Плохие тесты
```typescript
// ❌ Ничего не тестирует
expect(true).toBe(true)

// ❌ Только проверяет вызов мока
expect(api.call).toHaveBeenCalled()  // без проверки результата

// ❌ Просто рендерит без проверки
render(<Component />)  // что проверяем-то?
```

### Хорошие тесты
```typescript
// ✅ Проверяет реальный результат
expect(посчитатьИтого(100, 0.2)).toBe(80)

// ✅ Проверяет реальное изменение состояния
корзина.добавить({ id: 1 })
expect(корзина.товары).toHaveLength(1)
```

## Правило моков

- **Unit:** мокаем БД, API, файловую систему, время
- **Integration:** реальная БД (тестовая), мокаем внешние сервисы
- **E2E:** всё реальное (sandbox для внешних сервисов)
- **Если >3 моков в unit-тесте** → скорее всего нужен integration-тест

## Избыточное тестирование

Тесты-дубликаты — трата времени и обуза при поддержке.

Признаки:
- То же поведение проверяется и unit, и integration тестом без доп. пользы
- E2E тест проверяет только то, что уже покрыто unit-тестами
- Несколько тест-файлов тестируют одну функцию с одинаковыми сценариями

**Правило:** если удаление теста не снижает уверенность — тест лишний.

## Покрытие

- Минимум **80%** покрытия для бизнес-логики
- Тесты для всех публичных функций
- Edge cases покрыты
- Не гнаться за 100% — фокус на ценности теста

## Адаптивность по размеру задачи

| Уровень | S (Quick Fix) | M (Feature) | L (Epic) |
|---------|--------------|-------------|----------|
| Smoke | — | — | ✅ (при новом проекте) |
| Unit | если есть логика | ✅ | ✅ |
| Integration | — | ✅ (если API/БД) | ✅ |
| E2E | — | по запросу | ✅ (критичные) |
