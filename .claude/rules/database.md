# Правила работы с базой данных

## Миграции

### Общие правила
- ✅ ВСЕГДА использовать систему миграций (Alembic / Prisma Migrate / Django migrations)
- ✅ Миграции — часть кода, коммитятся в git
- ✅ Каждая миграция — атомарная операция (одно логическое изменение)
- ❌ НИКОГДА не редактировать старые миграции
- ❌ НИКОГДА не применять миграции вручную на проде без проверки

### Перед созданием миграции
1. Проверь текущее состояние: `alembic current` / `prisma migrate status`
2. Убедись что нет незаппланированных миграций
3. Проверь совместимость с существующими данными
4. Если удаляешь колонку/таблицу — сначала убери зависимости в коде

### Naming Convention
```
# Alembic
YYYY_MM_DD_HHMM_описание.py
# Пример: 2026_02_24_1430_добавить_таблицу_заказов.py

# Prisma
Описание на русском в migrate dev --name
# Пример: prisma migrate dev --name "добавить_таблицу_заказов"
```

### Rollback
- Каждая миграция ДОЛЖНА иметь downgrade/rollback
- Протестируй rollback ПЕРЕД деплоем: `alembic downgrade -1` → `alembic upgrade head`
- Для необратимых изменений (удаление данных) — предупредить пользователя

## Безопасные запросы

### ОБЯЗАТЕЛЬНО
```python
# ✅ Параметризованные запросы
cursor.execute("SELECT * FROM users WHERE id = %s", (user_id,))

# ✅ ORM
user = await User.get(id=user_id)

# ✅ SQLAlchemy
stmt = select(User).where(User.id == user_id)
```

### ЗАПРЕЩЕНО
```python
# ❌ Конкатенация строк
cursor.execute(f"SELECT * FROM users WHERE id = {user_id}")

# ❌ format()
query = "SELECT * FROM users WHERE name = '{}'".format(name)
```

## Схема данных

### Именование
- Таблицы: `snake_case`, множественное число (`пользователи`, `заказы`)
- Колонки: `snake_case` (`имя_пользователя`, `дата_создания`)
- Индексы: `idx_таблица_колонка` (`idx_заказы_дата_создания`)
- Foreign key: `fk_таблица_ссылка` (`fk_заказы_пользователь`)

### Обязательные поля
Каждая таблица ДОЛЖНА содержать:
- `id` — первичный ключ (BigInt auto-increment или UUID)
- `created_at` — дата создания (timestamp with timezone, default NOW())
- `updated_at` — дата обновления (timestamp with timezone, auto-update)

### Индексы
- ✅ Индекс на каждый foreign key
- ✅ Индекс на поля в WHERE/ORDER BY при >10K записей
- ✅ Составной индекс для частых комбинаций в WHERE
- ❌ Не создавать индексы "на всякий случай" — каждый индекс замедляет запись

## Производительность

### N+1 проблема
```python
# ❌ N+1 запросов
for order in orders:
    user = await User.get(id=order.user_id)  # запрос на каждый заказ!

# ✅ Один запрос с JOIN
orders = await Order.all().prefetch_related('user')
```

### Пагинация
- ✅ ВСЕГДА использовать пагинацию для списков
- ✅ LIMIT + OFFSET для простых случаев
- ✅ Cursor-based пагинация для больших объёмов (>100K записей)
- ❌ НИКОГДА не `SELECT * FROM table` без LIMIT

### Транзакции
```python
# ✅ Атомарные операции в транзакции
async with db.transaction():
    await списать_баланс(user_id, сумма)
    await создать_заказ(user_id, товары)
    await обновить_склад(товары)
```

## Бэкапы

### Перед деструктивными операциями
- Перед DROP TABLE / DROP COLUMN — бэкап таблицы
- Перед массовым UPDATE/DELETE — бэкап затронутых данных
- Формат: `pg_dump -t таблица > backup_YYYYMMDD.sql`

### На проде
- Автоматические бэкапы — минимум раз в день
- Проверять восстановление — минимум раз в месяц
- Хранить бэкапы минимум 30 дней

## Чувствительные данные

### PII (персональные данные)
- ✅ Документировать все поля с PII в architecture.md
- ✅ Шифровать пароли (bcrypt/argon2)
- ✅ Логировать доступ к PII
- ❌ НЕ хранить номера карт (используй токенизацию)
- ❌ НЕ выводить PII в логи

### Soft Delete
Для таблиц с пользовательскими данными:
```sql
-- Вместо DELETE
UPDATE users SET deleted_at = NOW() WHERE id = 123;

-- При выборке
SELECT * FROM users WHERE deleted_at IS NULL;
```

## Чеклист перед деплоем миграции

- [ ] Миграция протестирована локально
- [ ] Rollback работает
- [ ] Нет блокирующих операций на больших таблицах (ALTER TABLE с LOCK)
- [ ] Индексы создаются CONCURRENTLY (для PostgreSQL)
- [ ] Бэкап сделан
- [ ] Зависимости в коде уже обновлены (или backward-compatible)
