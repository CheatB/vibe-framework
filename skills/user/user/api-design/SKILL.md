# API Design

## Принципы

1. **Ресурсо-ориентированный** — URL описывает ресурс, метод — действие
2. **Предсказуемый** — одинаковая структура ответов везде
3. **Версионированный** — /api/v1/ с первого дня
4. **Документированный** — OpenAPI/Swagger из кода

## Структура URL

```
GET    /api/v1/пользователи          — список
GET    /api/v1/пользователи/:id      — один
POST   /api/v1/пользователи          — создать
PUT    /api/v1/пользователи/:id      — обновить (полностью)
PATCH  /api/v1/пользователи/:id      — обновить (частично)
DELETE /api/v1/пользователи/:id      — удалить
```

### Правила URL
- Множественное число для коллекций (`/заказы`, не `/заказ`)
- Вложенность макс 2 уровня (`/пользователи/:id/заказы`)
- Фильтрация через query params (`?статус=активный&лимит=20`)
- Без глаголов в URL (действие = HTTP метод)

## Формат ответа

### Успех
```json
{
  "data": { ... },
  "meta": {
    "page": 1,
    "per_page": 20,
    "total": 150
  }
}
```

### Ошибка
```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Невалидный email",
    "details": [
      {
        "field": "email",
        "message": "Некорректный формат email"
      }
    ]
  }
}
```

### HTTP коды
| Код | Когда |
|-----|-------|
| 200 | Успех (GET, PUT, PATCH) |
| 201 | Создано (POST) |
| 204 | Без контента (DELETE) |
| 400 | Ошибка валидации |
| 401 | Не авторизован |
| 403 | Нет прав |
| 404 | Не найдено |
| 409 | Конфликт (дубликат) |
| 422 | Unprocessable Entity |
| 429 | Слишком много запросов |
| 500 | Ошибка сервера |

## Пагинация

### Offset-based (простые случаи)
```
GET /api/v1/заказы?page=2&per_page=20
```

### Cursor-based (большие объёмы, >100K)
```
GET /api/v1/заказы?cursor=abc123&limit=20
```

Ответ содержит `next_cursor` для следующей страницы.

## Фильтрация и сортировка

```
GET /api/v1/заказы?статус=оплачен&сумма_от=1000&sort=-дата_создания
```

- Фильтры: `поле=значение`
- Диапазоны: `поле_от`, `поле_до`
- Сортировка: `sort=поле` (asc), `sort=-поле` (desc)
- Поиск: `search=текст`

## Аутентификация

```
Authorization: Bearer <jwt_token>
```

- JWT в заголовке Authorization
- Refresh token через httpOnly cookie
- Для API-интеграций: API key в заголовке `X-API-Key`

## Валидация

Каждый эндпоинт ОБЯЗАТЕЛЬНО валидирует входные данные:

```python
# Python + Pydantic
class СоздатьЗаказ(BaseModel):
    товары: list[ТоварВЗаказе]
    адрес_доставки: str = Field(min_length=5, max_length=500)
    комментарий: str | None = Field(None, max_length=1000)
```

```typescript
// TypeScript + Zod
const СхемаЗаказа = z.object({
  товары: z.array(СхемаТовара).min(1),
  адресДоставки: z.string().min(5).max(500),
  комментарий: z.string().max(1000).optional(),
})
```

## Rate Limiting

```
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 95
X-RateLimit-Reset: 1640000000
```

Стандартные лимиты:
- Публичные: 60 req/min
- Авторизованные: 100 req/min
- Админские: 300 req/min

## Версионирование

- `/api/v1/` — с первого дня
- Новая версия при breaking changes
- Старая версия поддерживается минимум 6 месяцев
- Deprecation через заголовок `Sunset: <date>`

## Документация

- OpenAPI/Swagger генерируется из кода (не пишется вручную)
- Каждый эндпоинт: описание, параметры, примеры ответов
- Доступна по `/api/docs` или `/api/swagger`

## Чеклист нового эндпоинта

- [ ] URL ресурсо-ориентированный
- [ ] HTTP метод правильный
- [ ] Валидация входных данных
- [ ] Правильные HTTP коды ответов
- [ ] Единый формат ответа (data + error)
- [ ] Авторизация (если нужна)
- [ ] Rate limiting
- [ ] Тест (integration test обязателен)
- [ ] Документация (OpenAPI)

## Приоритет при конфликтах

- Безопасность (валидация, auth) > удобство
- Консистентность формата > оптимальность отдельного эндпоинта
- Если проект уже имеет свой стиль API → следуй ему, не навязывай новый
