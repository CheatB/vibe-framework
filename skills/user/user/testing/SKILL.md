# Testing Master

## Тестовая пирамида

```
        /\
       /E2E\        ← Мало (3-5 критичных сценариев)
      /------\
     /Интегр.\      ← Средне (все эндпоинты + БД)
    /----------\
   /   Unit     \   ← Много (вся бизнес-логика)
  /--------------\
 /    Smoke      \  ← Минимум (1-2 базовых теста)
/------------------\
```

## Когда какие тесты

### Smoke-тесты
**Цель:** Проверить что проект вообще запускается.
- Тестовый фреймворк настроен
- Env vars доступны
- Базовые импорты работают
- **Когда:** один раз при настройке проекта

**Настраиваешь smoke-тесты?** Читай [smoke-tests.md](references/smoke-tests.md)

### Unit-тесты
**Цель:** Тестировать бизнес-логику изолированно.
- Функции с расчётами, валидацией, трансформацией
- Логика принятия решений (if/else)
- Обработка ошибок
- **Пропустить:** простые геттеры, однострочные изменения

**Пишешь unit-тесты?** Читай [unit-tests.md](references/unit-tests.md)

### Интеграционные тесты
**Цель:** Тестировать API, БД, внешние сервисы.
- Все API-эндпоинты (особенно POST/PUT/DELETE)
- Операции с БД (CRUD)
- Интеграции (платежи, email, webhooks)
- **Правило:** каждый эндпоинт + каждая запись в БД = тест

**Пишешь интеграционные?** Читай [integration-tests.md](references/integration-tests.md)

### E2E-тесты
**Цель:** Тестировать критичные пользовательские сценарии.
- Топ 3-5 критичных flows
- Большие фичи (>5 задач)
- Авторизация, платежи, core-фичи

**Пишешь E2E?** Читай [e2e-tests.md](references/e2e-tests.md)

## Фреймворк решений

### Нужны unit-тесты?
**ДА:** бизнес-логика, решения, трансформация данных, ошибки.
**НЕТ:** getter/setter, однострочное изменение, конфиг.

### Нужны интеграционные?
**ДА:** tech-spec требует, есть API, работа с БД, внешние сервисы.
**НЕТ:** tech-spec "None", чисто клиент, покрыто E2E.

### Нужны E2E?
**ДА:** >5 задач, критичный flow, breaking changes, запрос пользователя.
**НЕТ:** <3 задач, некритично, покрыто unit + integration.

## Ключевые принципы

1. Тесты СРАЗУ — в той же сессии, до перехода дальше
2. Тестируй поведение, не реализацию
3. Быстрые: unit=мс, integration=сек, E2E=мин
4. Изоляция: мокай внешнее в unit
5. Один тест = одна проверка
6. Понятные имена на русском
7. Чистое состояние: каждый тест стартует чисто
8. Каждый тест оправдывает себя — ловит уникальную ошибку

## Стратегия моков

| Уровень | Реальное | Мокаем |
|---------|---------|--------|
| Unit | Только тестируемая функция | БД, API, файлы, время |
| Integration | БД (тестовая), файлы | Внешние сервисы |
| E2E | Всё | Ничего (sandbox для внешних) |

**Если >3 моков → скорее всего нужен integration/E2E тест.**

## Качество тестов

### Плохой тест
```typescript
expect(true).toBe(true)                    // Ничего не проверяет
expect(api.call).toHaveBeenCalled()        // Только мок, не результат
render(<Component />)                       // Рендер без проверки
```

### Хороший тест
```typescript
expect(посчитатьИтого(100, 0.2)).toBe(80)  // Реальный результат
корзина.добавить({ id: 1 })
expect(корзина.товары).toHaveLength(1)       // Реальное изменение
```

## Приоритет при конфликтах

- Если tech-spec указывает конкретные тесты → следуй tech-spec
- Если rules/testing.md конфликтует со SKILL.md → SKILL.md приоритетнее (более детальный)
- Если пользователь просит пропустить тесты → предупредить, но выполнить
- Если покрытие <80% → не блокировать, но записать в техдолг
