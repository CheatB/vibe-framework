# Code Writing

Универсальные паттерны качественного кода. Проектно-специфичные паттерны — в project-knowledge/patterns.md.

## Структура файла

```
1. Импорты (сгруппированы: stdlib → внешние → внутренние)
2. Константы / Типы
3. Основная логика (публичные функции)
4. Вспомогательные функции (приватные)
5. Экспорт (если нужен)
```

## Размеры

- Файл > 500 строк → разбить на модули
- Функция > 50 строк → разбить на части
- Один файл = одна ответственность
- Класс > 10 методов → возможно нарушает SRP

## Именование (на русском где уместно)

```python
# ✅ Хорошо
максимум_попыток = 3
async def получить_пользователя(id_пользователя: int) -> Пользователь:
    """Получить пользователя по ID из базы данных."""

# ❌ Плохо
a = 3
async def get_user(id):
    pass
```

### Правила
- Переменные: `snake_case`, существительные (`список_заказов`, `текущий_пользователь`)
- Функции: `snake_case`, глагол + объект (`создать_заказ`, `отправить_уведомление`)
- Классы: `PascalCase` (`ОбработчикПлатежей`, `СервисАвторизации`)
- Константы: `UPPER_SNAKE_CASE` (`МАКС_ПОПЫТКИ`, `БАЗОВЫЙ_URL`)
- Булевы: вопрос (`активен`, `может_редактировать`, `имеет_доступ`)

## Обработка ошибок

### Принцип: fail fast, fail loud

```python
# ✅ Конкретные исключения
try:
    результат = await api.запрос(данные)
except ОшибкаПодключения as е:
    logger.error(f"Не удалось подключиться: {е}")
    raise
except ОшибкаВалидации as е:
    return Ответ(ошибка=str(е), код=400)

# ❌ Глотание ошибок
try:
    что_то_сделать()
except:
    pass  # Никогда так не делай!
```

### Правила
1. Ловить конкретные исключения, не `Exception`
2. Логировать ошибки перед обработкой
3. Не глотать ошибки (`except: pass`)
4. Кастомные исключения для бизнес-логики
5. Пользователю — понятное сообщение, в логи — полный traceback

## Паттерны

### Early Return
```python
# ✅ Ранний выход
async def обработать_заказ(заказ):
    if not заказ:
        return Ошибка("Заказ не найден")
    if заказ.оплачен:
        return Ошибка("Уже оплачен")
    if not заказ.товары:
        return Ошибка("Корзина пуста")

    # Основная логика — без вложенности
    return await провести_оплату(заказ)
```

### Guard Clauses
Валидация входных данных В НАЧАЛЕ функции, до основной логики.

### Dependency Injection
```python
# ✅ Зависимости через параметры
class СервисЗаказов:
    def __init__(self, бд: БазаДанных, платежи: СервисПлатежей):
        self.бд = бд
        self.платежи = платежи
```

### Repository Pattern
Отделяй работу с БД от бизнес-логики:
```
handlers/ → вызывает → services/ → вызывает → database/
(HTTP)       (логика)      (SQL/ORM)
```

## Документирование

```python
async def создать_пользователя(
    имя: str,
    email: str,
    роль: Роль = Роль.ОБЫЧНЫЙ
) -> Пользователь:
    """Создать нового пользователя в системе.

    Args:
        имя: Полное имя пользователя
        email: Email (будет провалидирован)
        роль: Роль в системе, по умолчанию обычный пользователь

    Returns:
        Созданный пользователь с присвоенным ID

    Raises:
        ОшибкаВалидации: Если email невалидный или уже занят
    """
```

## Чеклист перед коммитом

- [ ] Нет magic numbers (всё в константах)
- [ ] Нет хардкода (env vars для конфигов)
- [ ] Нет дублирования (DRY)
- [ ] Функции < 50 строк
- [ ] Понятные имена переменных
- [ ] Docstrings на русском
- [ ] Обработка ошибок (не `except: pass`)
- [ ] Нет `console.log` / `print()` для отладки
