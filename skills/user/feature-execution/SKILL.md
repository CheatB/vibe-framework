# Feature Execution

Полный pipeline реализации фичи от получения задачи до завершения.

## Workflow

### 1. Получение задачи

```
━━━ 🎯 РЕЖИМ: Feature ━━━
Задача: [описание]
Оценка: ~[время]
```

1. Прочитай описание задачи (из бэклога / tasks.md / от пользователя)
2. Определи размер: S (<30мин) / M (1-4ч) / L (дни)
3. Для S → сразу к шагу 4 (пропускаем планирование)
4. Для M/L → продолжай по порядку

### 2. Контекст (M/L)

1. Прочитай project-knowledge (project.md, architecture.md, patterns.md)
2. Проверь DECISIONS.md — есть ли релевантные решения
3. Проверь бэклог — есть ли связанные задачи
4. Grep usages изменяемых файлов — кто их использует

### 3. Планирование (M/L)

```
━━━ 📋 ФАЗА 2: ПЛАН ━━━
Блоков: [N] | Задач: [M] | Оценка: ~[время]
```

1. Разбей на блоки (логические группы задач)
2. Определи зависимости между блоками
3. Для каждого блока: файлы для создания/изменения, тесты
4. Покажи план пользователю → жди одобрения

### 4. Выполнение

```
━━━ 🔨 ФАЗА 3: ВЫПОЛНЕНИЕ ━━━
Блок [1/N]: [название]
```

Для каждого блока:
1. **TDD Anchor:** Напиши тест-заглушку (описание теста + skip)
2. **Код:** Напиши реализацию
3. **Тесты:** Раскомментируй тесты, добейся green
4. **Anti-Mirage:** Проверь импорты, вызовы, env vars (rules/anti-mirage.md)
5. **Quality Gate 2:** Ревью кода молча (rules/quality-gates.md)

Между блоками — чекпоинт:
```
━━━ ✅ ЧЕКПОИНТ: Блок [N] завершён ━━━
Сделано: [что]
Следующий: [что]
⏸️ Продолжаем?
```

### 5. Тестирование

```
━━━ 🧪 ФАЗА 4: ТЕСТИРОВАНИЕ ━━━
```

1. Запусти все тесты проекта
2. Проверь покрытие (>80% для бизнес-логики)
3. Если тесты падают → исправь (макс 3 попытки, потом покажи ошибку)

### 6. Завершение

Выполни /done:
1. Коммит с осмысленным сообщением
2. Обнови project-knowledge если нужно
3. Обнови бэклог
4. Запиши решения в DECISIONS.md если были
5. Итоговый отчёт пользователю

## Размеры и адаптация

| Шаг | S (Quick Fix) | M (Feature) | L (Epic) |
|-----|--------------|-------------|----------|
| Контекст | grep usages | project-knowledge | full audit |
| Планирование | — | план блоков | plan + ревью архитектуры |
| TDD Anchors | — | ✅ | ✅ |
| Чекпоинты | — | между блоками | между блоками + deep review |
| Quality Gate | anti-mirage | full | full + security |
| DECISIONS.md | — | если были | обязательно |

## Правила

1. **Макс 3 итерации** на исправление ошибки. После — показать пользователю.
2. **Silent Review** — ревью молча, пользователю только финальный результат.
3. **Чекпоинты обязательны** — не пропускать между блоками.
4. **Anti-Mirage всегда** — даже для S-задач, это дёшево и ценно.

## Приоритет при конфликтах

- Пользователь > процесс (если просит пропустить шаг — пропускаем с предупреждением)
- Безопасность > скорость (security review не пропускаем для sensitive кода)
- Рабочий код > идеальный код (better done than perfect)
