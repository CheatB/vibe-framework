# Project Knowledge

Скилл предоставляет доступ к документации проекта: как устроен, как писать код, как разрабатывать фичи.

## Когда активировать

- Понять архитектуру, стек, модель данных
- Узнать паттерны кода, git workflow, подход к тестированию
- Проверить настройки деплоя, мониторинга
- Применить UX-гайдлайны и дизайн-систему
- Принять решение, согласованное со стандартами проекта

## Основные справочники

Вся документация в `references/`:

- **[project.md](references/project.md)** — обзор проекта, цель, аудитория, ключевые фичи, границы scope
- **[architecture.md](references/architecture.md)** — стек (+ почему), структура, зависимости, интеграции, data flow, модель данных (схема, миграции, чувствительные данные)
- **[patterns.md](references/patterns.md)** — проектные паттерны, git workflow (ветки, тестирование, security gates), методы верификации, бизнес-правила
- **[deployment.md](references/deployment.md)** — платформа, env vars, CI/CD триггеры, rollback, мониторинг (логи, ошибки, health checks)

## Опциональные справочники

- **[ux-guidelines.md](references/ux-guidelines.md)** — язык интерфейса, tone of voice, глоссарий, текстовые паттерны, дизайн-система (только для проектов со значительным UI)
- **{domain}.md** — домен-специфичные файлы (bot.md, vault.md, api.md)

## Как использовать

Читай конкретные справочники по задаче:

| Задача | Что читать |
|--------|-----------|
| Начинаешь фичу | project.md, architecture.md, patterns.md |
| Меняешь БД | architecture.md (секция Data Model) |
| Работаешь с UI | ux-guidelines.md |
| Деплоишь | deployment.md |
| Создаёшь ветку/PR | patterns.md (Git Workflow) |
| Разбираешь логи/ошибки | deployment.md (Monitoring) |
| Доменная логика | patterns.md (Business Rules) |

## Workflow обновления

### При /done (завершение фичи)
1. Проверь — изменилась ли архитектура, стек, паттерны?
2. Если да → обнови соответствующий справочник
3. Если принято важное решение → запиши в DECISIONS.md

### При /docs audit
1. Прочитай все справочники + CLAUDE.md
2. Найди устаревшее, дублирование, bloat
3. Исправь по правилам из rules/documentation.md

## Приоритет при конфликтах

- project-knowledge > общие rules (проектная специфика важнее)
- patterns.md содержит ТОЛЬКО проектные паттерны, НЕ универсальные
- Если справочник устарел → обновить, а не игнорировать
- CLAUDE.md — минимальный указатель, НЕ контейнер информации
