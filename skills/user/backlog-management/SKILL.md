# Backlog Management

## Структура бэклога

Файл `backlog.md` в корне проекта (или путь из CLAUDE.md):

```markdown
# Бэклог проекта

## 🔥 Критичные (блокеры)
- [ ] [#1] Описание — S/M/L

## 📋 Высокий приоритет
- [ ] [#2] Описание — S/M/L

## 📝 Средний приоритет
- [ ] [#3] Описание — S/M/L

## 💡 Низкий (nice to have)
- [ ] [#4] Описание — S/M/L

## ✅ Выполнено
- [x] [#0] Описание — дата
```

## Workflow: показать бэклог

1. Найди backlog.md (корень проекта или путь из CLAUDE.md)
2. Покажи сводку: всего задач, по приоритетам, что в работе, топ-3 следующих
3. Если бэклога нет — предложи создать

## Workflow: добавить задачу

1. Пойми что за задача (описание от пользователя или из контекста)
2. Определи приоритет (критичный / высокий / средний / низкий)
3. Оцени размер (S <30мин / M 1-4ч / L дни)
4. Присвой следующий #ID
5. Добавь в правильную секцию
6. Покажи результат

Если пользователь описал в свободной форме → определи приоритет и размер сам, покажи для подтверждения.

## Workflow: следующая задача

1. Прочитай бэклог
2. Выбери задачу с наивысшим приоритетом:
   - Не заблокирована другими
   - Не в работе
   - Предпочитай S/M (quick wins)
3. Покажи пользователю для подтверждения
4. При подтверждении → отметь "в работе", начни выполнение

## Workflow: приоритизация

1. Прочитай весь бэклог
2. Оцени по критериям:
   - Бизнес-ценность (что даст пользователям)
   - Зависимости (что разблокирует)
   - Усилие (quick wins в первую очередь)
3. Предложи пересортировку
4. Примени после одобрения

## Интеграции

### С Deep Interview (/new-project)
Неотвеченные вопросы → задачи в бэклоге с тегом `[вопрос]`.
При /backlog next → вопросы видны в контексте связанных задач.

### С /done
Завершённые задачи → переносятся в "✅ Выполнено" с датой.
Новые задачи из фичи → добавляются в бэклог автоматически.

### С tasks.md
tasks.md = оперативный план текущей работы (волны, блоки).
backlog.md = стратегический список всех задач проекта.
Не дублировать. При начале работы задача уходит из бэклога в tasks.md.

## Приоритет при конфликтах

- Пользователь > алгоритм приоритизации (если просит конкретную задачу — делаем)
- Блокеры > всё остальное
- Quick wins (S) > долгие задачи (L) при равном приоритете
- Если бэклог пуст или нет приоритизации → спроси пользователя

## Пример

```
📋 Бэклог проекта Project1

Всего: 12 задач | 🔥 2 | 📋 4 | 📝 4 | 💡 2
В работе: [#7] Добавить авторизацию — M

Топ-3 следующих:
1. [#3] Исправить баг с оплатой — S (🔥 критичный)
2. [#5] API для выгрузки отчётов — M (📋 высокий)
3. [#8] Кеширование Redis — M (📋 высокий)
```
