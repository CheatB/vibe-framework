# /migrate — Управление миграциями базы данных

Команда для создания, проверки и применения миграций БД.

## Использование

```
/migrate              — показать статус миграций
/migrate create       — создать новую миграцию
/migrate check        — проверить миграцию перед применением
/migrate apply        — применить миграции
/migrate rollback     — откатить последнюю миграцию
```

## /migrate (без аргументов) — Статус

1. Определи ORM/инструмент миграций проекта:
   - Alembic (Python/SQLAlchemy)
   - Prisma Migrate (Node.js)
   - Django migrations
   - Tortoise ORM (Python)
   - Ручные SQL скрипты
2. Покажи текущее состояние:
   ```
   📊 Статус миграций

   Инструмент: Alembic
   Текущая ревизия: abc123 (добавить_таблицу_заказов)
   Последняя доступная: def456 (добавить_индексы)
   Непримённые: 1
   ```

## /migrate create — Создание миграции

1. Спроси что нужно изменить (или прими описание)
2. Сгенерируй миграцию:
   ```bash
   # Alembic
   alembic revision --autogenerate -m "описание_на_русском"

   # Prisma
   prisma migrate dev --name "описание_на_русском"
   ```
3. **ОБЯЗАТЕЛЬНО проверь сгенерированную миграцию:**
   - Все изменения корректны?
   - Нет лишних/пропущенных операций?
   - Есть downgrade/rollback?
   - Нет блокирующих операций на больших таблицах?
4. Покажи пользователю для ревью

## /migrate check — Проверка перед применением

Запусти чеклист из rules/database.md:

```
🔍 Проверка миграции: [название]

Безопасность:
✅ Нет DROP TABLE/COLUMN без бэкапа
✅ Нет блокирующих ALTER на больших таблицах
✅ Индексы создаются CONCURRENTLY

Совместимость:
✅ Backward-compatible (код работает и до, и после)
✅ Зависимости в коде обновлены
✅ Данные не теряются

Rollback:
✅ downgrade() определён
✅ downgrade() протестирован

Производительность:
⚠️ Добавлен индекс на таблицу >100K записей — может занять время
```

## /migrate apply — Применение

1. Запусти /migrate check
2. Если есть ❌ — остановись, покажи проблемы
3. Если всё ✅ или ⚠️ — спроси подтверждение:
   ```
   Применить миграцию "[название]"?
   ⚠️ Предупреждение: добавление индекса может занять ~30 сек
   ```
4. Примени:
   ```bash
   alembic upgrade head
   # или
   prisma migrate deploy
   ```
5. Проверь результат: `alembic current`

## /migrate rollback — Откат

1. Покажи текущую миграцию
2. Предупреди о последствиях:
   ```
   ⚠️ Откат миграции "[название]"

   Будет отменено:
   - Удалена таблица orders
   - Удалён индекс idx_orders_date

   Это НЕОБРАТИМО для данных в этих таблицах.
   Продолжить?
   ```
3. При подтверждении:
   ```bash
   alembic downgrade -1
   ```
4. Проверь результат

## Безопасность

### Опасные операции (требуют подтверждения)
- DROP TABLE / DROP COLUMN
- ALTER TABLE с потерей данных (изменение типа колонки)
- DELETE / TRUNCATE в миграции
- Удаление индексов на продакшене

### Рекомендации для больших таблиц (>100K записей)
- CREATE INDEX CONCURRENTLY вместо CREATE INDEX
- ALTER TABLE в отдельной миграции
- Добавление NOT NULL через default → backfill → constraint
- Предупреждать пользователя о времени выполнения

## Интеграция

- Перед деплоем (/deploy) — автоматический /migrate check
- При /new-project — создание начальной миграции
- В rules/database.md — полные правила работы с БД
