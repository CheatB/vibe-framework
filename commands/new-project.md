
# Создание нового проекта — Vibe Framework

Полный workflow от идеи до рабочего продукта.
Ключевое: ГЛУБОЧАЙШАЯ проработка ДО написания кода. Лучше потратить 2 часа на проработку, чем 2 дня на переделку.

---

## ФАЗА 0: Загрузка контекста

Загрузить всё, что есть. Спросить у пользователя:

1. **Есть ли документы?** (заметки, описания, наброски) — прочитать ВСЁ
2. **Название проекта** (латиницей)
3. **Расскажи о себе и зачем тебе это** — роль, бизнес-контекст, мотивация, боль которую решаем
4. **Опиши задачу как ты её понимаешь** — своими словами, без фильтров, как есть
5. **Для кого это?** — целевая аудитория, их проблемы
6. **Где будет жить?** — VPS, домен
7. **Стек** — если знаешь; если нет — предложу после проработки

Записать в `docs/context.md`. Этот файл — фундамент всего.

## ФАЗА 1: Brainstorming

### Субагент: Бизнес-аналитик

Делегировать субагенту с ролью бизнес-аналитика. Промпт субагенту:

> Ты — опытный бизнес-аналитик. Твоя задача — вытащить из заказчика АБСОЛЮТНО ВСЁ, что он знает и чего не знает о проекте. Не останавливайся на поверхности. Копай глубже. Задавай follow-up вопросы. Если ответ расплывчатый — уточняй. Если заказчик говорит "хз" — предложи 2-3 варианта с плюсами/минусами и спроси какой ближе.
>
> Прочитай docs/context.md для контекста.
>
> Задавай вопросы группами по 3-5 штук. После каждой группы ответов — задавай follow-up. Продолжай пока не будешь уверен, что покрыл ВСЁ.

### Что должен покрыть бизнес-аналитик:

**Пользователи и их мир:**
- Кто конкретно будет пользоваться? Составить портрет каждого типа пользователя
- Какой у них уровень технической грамотности?
- Как они решают эту проблему СЕЙЧАС, без нашего продукта?
- Что их бесит в текущем решении?
- В каком контексте они будут использовать продукт? (на ходу с телефона, за компом, в офисе)
- Как часто будут пользоваться? Раз в день, раз в неделю?
- Что заставит их БРОСИТЬ пользоваться?

**Сценарии от начала до конца:**
- Пройти каждый сценарий пошагово: пользователь открывает → делает → видит → уходит
- Что происходит ДО того, как пользователь открывает продукт?
- Что происходит ПОСЛЕ?
- Happy path — когда всё идёт хорошо
- Sad path — когда что-то идёт не так
- Edge cases — странные, но возможные ситуации

**Функционал — границы:**
- Что система ОБЯЗАНА делать? (без этого продукт бесполезен)
- Что ХОЧЕТСЯ, но можно потом?
- Что система ТОЧНО НЕ делает? (чёткая граница scope)
- Есть ли похожие продукты? Чем мы отличаемся?

**Бизнес-логика:**
- Как зарабатываем? Модель монетизации
- Какие числа нас интересуют? (метрики)
- Как поймём что продукт успешен? Конкретные критерии по SMART
- Есть ли дедлайны или привязки к событиям?

**Интеграции и зависимости:**
- С какими внешними системами работаем?
- Есть ли уже аккаунты, ключи, доступы?
- Какие ограничения у внешних API?
- Что если внешний сервис ляжет?

**Данные:**
- Какие данные собираем и храним?
- Какие данные НИКОГДА не храним?
- Нужно ли импортировать данные откуда-то?
- Приватность, GDPR, персональные данные?

**Риски и страхи:**
- Что может пойти не так?
- Чего ты больше всего боишься в этом проекте?
- Были ли попытки сделать что-то подобное раньше? Почему не получилось?

### Правила:
- НЕ останавливаться после первого раунда вопросов
- После каждой группы ответов — формулировать что понял и задавать follow-up
- Если ответ "хз" — предложить варианты, обсудить, выбрать
- Продолжать пока ВСЕ категории не покрыты
- Записывать ВСЁ в `docs/brainstorm.md`

## ФАЗА 2: User-Spec (понятийное техзадание)

### Субагент: Бизнес-аналитик (продолжение)

Тот же субагент берёт результаты brainstorming и пишет user-spec.

> На основе brainstorming напиши понятийное техзадание. ПРОСТЫМ языком, как будто объясняешь проект человеку, который вообще не разбирается в технологиях. Никаких технических терминов. Каждый сценарий — пошагово, как инструкция для ребёнка.

### Структура user-spec (`docs/user-spec.md`):

```markdown
# [Название] — Что мы делаем и зачем

## В одном предложении
[Что за продукт и какую проблему решает]

## Для кого это
[Портрет пользователя, его боль, его мир]

## Как это работает

### Сценарий 1: [название]
**Контекст:** [в какой ситуации пользователь приходит]
1. Пользователь делает [что]
2. Система показывает [что]
3. Пользователь выбирает [что]
4. Система делает [что]
5. Пользователь получает [результат]
**Если что-то пошло не так:** [что видит пользователь]

### Сценарий 2: [название]
[...]

## Что входит в первую версию (MVP)
Для каждой функции:
- **[Функция]** — зачем нужна, как работает простыми словами

## Что НЕ делаем сейчас
- **[Функция]** — почему не сейчас, когда вернёмся

## О чём надо подумать (риски и нюансы)
- [Риск 1] — почему это важно, что будем делать
- [Риск 2]

## Как поймём что сделали нормально
Конкретные критерии по SMART:
- [Критерий 1] — что именно измеряем, какое значение = успех
- [Критерий 2]
- [Критерий 3]
```

### Правила user-spec:
- НОЛЬ технических терминов. Никаких "API", "эндпоинт", "база данных"
- Каждый сценарий — полностью, от входа до выхода, включая ошибки
- Критерии приёмки — конкретные и измеримые
- Если нельзя объяснить бабушке — переписать

**✅ Контрольная точка: обсуждаем, правим, согласовываем user-spec. Не двигаемся дальше пока пользователь не скажет "ок, всё верно".**

## ФАЗА 3: Tech-Spec (техническое задание)

### Субагент: Системный аналитик / Архитектор

Делегировать НОВОМУ субагенту с ролью системного аналитика и архитектора.

> Ты — системный аналитик и архитектор с 15-летним опытом. Твоя задача — превратить понятийное техзадание (user-spec) в МАКСИМАЛЬНО подробное техническое задание. Задротский уровень. Каждое решение обосновано. Каждый edge case прописан. Каждый формат данных определён.
>
> Прочитай docs/context.md, docs/brainstorm.md и docs/user-spec.md.
>
> Пиши tech-spec на десятки страниц. Лучше 50 страниц, которые покрывают ВСЁ, чем 5 страниц с дырами.

### Структура tech-spec (`docs/tech-spec.md`):

```markdown
# [Название] — Technical Specification

## 1. Обзор системы
- Назначение
- Ключевые требования (функциональные и нефункциональные)
- Ограничения и допущения

## 2. Архитектура
- Общая схема (компоненты и связи между ними)
- Обоснование выбора архитектуры
- Диаграмма компонентов
- Стек технологий — для каждой технологии: ЧТО, ПОЧЕМУ, какие альтернативы рассматривались
- Схема деплоя

## 3. Структура проекта
- Дерево файлов с описанием назначения КАЖДОЙ папки и файла
- Конвенции именования

## 4. Модели данных
Для каждой сущности:
- Название и назначение
- Все поля: имя, тип, обязательность, значение по умолчанию, валидация
- Связи с другими сущностями
- Индексы и почему
- Примеры данных
- SQL миграции

## 5. API / Endpoints / Handlers / Команды
Для КАЖДОГО:
- Путь / команда / триггер
- Метод (GET/POST/etc) или тип события
- Входные параметры: имя, тип, обязательность, валидация, пример
- Логика обработки — пошагово
- Выходные данные: формат, поля, примеры
- Ошибки: код, сообщение, когда возникает, как обрабатываем
- Rate limits
- Авторизация

## 6. Бизнес-логика
Для каждого процесса:
- Входные условия
- Пошаговый алгоритм
- Выходные условия
- Граничные случаи и как обрабатываем
- Формулы расчётов (если есть)

## 7. Интеграции
Для КАЖДОЙ внешней системы:
- Название сервиса и версия API
- Аутентификация
- Список используемых эндпоинтов
- Формат запросов (с примерами)
- Формат ответов (с примерами)
- Обработка ошибок API
- Rate limits и throttling стратегия
- Retry-политика
- Fallback если сервис недоступен
- Стоимость использования

## 8. Безопасность
- Аутентификация: механизм, токены, сессии
- Авторизация: роли, права, проверки
- Валидация ВСЕХ входных данных
- Защита от: SQL injection, XSS, CSRF, path traversal
- Хранение секретов
- Логирование действий (аудит)

## 9. Обработка ошибок
- Классификация ошибок (пользовательские, системные, внешние)
- Стратегия для каждого класса
- Retry-политики с backoff
- Graceful degradation
- Уведомления об ошибках (куда, кому)

## 10. Инфраструктура
- Docker: Dockerfile, docker-compose.yml — полные конфиги
- Nginx: конфигурация с пояснениями
- SSL: получение и обновление
- Переменные окружения: полный список с описаниями
- Мониторинг и health checks
- Логирование: что логируем, куда, формат
- Бэкапы: что, как часто, куда, как восстанавливать

## 11. Тестирование
- Стратегия тестирования
- Что покрываем unit-тестами
- Что покрываем integration-тестами
- Тестовые данные и фикстуры
- Как запускать тесты

## 12. Производительность
- Ожидаемая нагрузка
- Узкие места и как решаем
- Кэширование (если нужно)
- Оптимизация запросов к БД
```

### Правила tech-spec:
- МАКСИМАЛЬНАЯ детализация. Если можно описать подробнее — описать подробнее
- Каждое техническое решение — с обоснованием ПОЧЕМУ именно так
- Все edge cases прописаны
- Примеры данных для каждой сущности и API
- Формулы, алгоритмы — расписать пошагово
- Если есть несколько вариантов решения — описать все, выбрать лучший, объяснить почему

**✅ Контрольная точка: обсуждаем, правим, согласовываем tech-spec. Это САМЫЙ важный документ. Не двигаемся дальше без полного согласования.**

## ФАЗА 4: Task Breakdown (декомпозиция)

### Субагент: Системный аналитик (продолжение)

> Разбей tech-spec на АТОМАРНЫЕ задачи. Каждая задача — максимально простая и конкретная. 5-15 минут работы. Для каждой задачи — своё мини-ТЗ и чёткие критерии приёмки.

### Формат (`docs/tasks.md`):

```markdown
# Задачи проекта [Название]

## Блок 1: [название] (~X мин)

### DoD блока 1:
- [ ] Все задачи блока выполнены
- [ ] Нет заглушек (TODO, FIXME, pass, NotImplementedError)
- [ ] Тесты блока зелёные
- [ ] Нет дублирования логики с предыдущими блоками
- [ ] Контракты между модулями согласованы (интерфейсы, типы, форматы)
- [ ] Код отформатирован (black/prettier)
- [ ] Нет console.log/print для отладки
- [ ] AI_NOTES блока написан

### Задача 1.1: [название]
**Что:** [конкретно что сделать]
**Зачем:** [какую часть user-spec/tech-spec реализует]
**Как:**
1. [конкретный шаг]
2. [конкретный шаг]
3. [конкретный шаг]
**Файлы:** [создать/изменить]
**Зависимости:** [задачи, которые должны быть сделаны раньше]
**Критерии приёмки:**
- [ ] [конкретная проверка: "запустить X, должен вернуть Y"]
- [ ] [конкретная проверка]
- [ ] [конкретная проверка]
**Оценка:** ~X мин

### Задача 1.2: [название]
[...]

## Блок 2: [название] (~X мин)
### DoD блока 2:
[...]
```

### Правила декомпозиции:
- Задача = 5-15 минут, МАКСИМУМ 20
- Если задача больше 20 минут — разбить ещё
- Критерии приёмки КОНКРЕТНЫЕ: "запустить pytest, 0 ошибок", "curl /api/health возвращает 200"
- Зависимости между задачами явные
- Блоки по 30-60 минут
- Каждый блок включает DoD (Definition of Done)

**✅ Контрольная точка: пользователь одобряет список задач.**

## ФАЗА 5: Выполнение

Работать по задачам из `docs/tasks.md` последовательно.

### Цикл на каждую задачу:
1. Прочитать ТЗ задачи
2. Написать тест (TDD)
3. Написать код
4. Проверить критерии приёмки
5. Отметить задачу как выполненную в `docs/tasks.md`

### После КАЖДОГО блока — обязательно:
1. Проверить DoD блока (все пункты из tasks.md)
2. Написать `docs/ai-notes/block-N.md`
3. Показать пользователю результат

### AI_NOTES — журнал решений

После завершения каждого блока задач — записать заметку в `docs/ai-notes/block-N.md`.
Это решает главную проблему вайбкодинга: на дистанции LLM теряет контекст, дублирует логику, расходит стиль, оставляет заглушки.

Формат `docs/ai-notes/block-N.md`:

```markdown
# Блок N: [название]

## Что сделано
- [конкретно что реализовано]

## Решения и ПОЧЕМУ
- [решение 1] — почему именно так, какие альтернативы отвергли
- [решение 2] — ...

## Контракты
- [модуль A] → [модуль B]: формат данных, интерфейс
- [внешний API] → [наш сервис]: формат запроса/ответа

## Риски и технический долг
- [что может сломаться]
- [что упростили сейчас и нужно доделать потом]

## Как проверить
- [команда для тестов]
- [curl для endpoint]

## Контекст для следующего блока
- [что нужно знать чтобы продолжить]
- [какие файлы трогать, какие не трогать]
```

### Зачем AI_NOTES:
- **Compaction** — при сжатии контекста Клод читает последний AI_NOTES и восстанавливает контекст
- **Передача субагенту** — координатор передаёт AI_NOTES предыдущего блока новому разработчику
- **Возобновление работы** — после перерыва читаешь AI_NOTES и сразу в курсе
- **Защита от дрифта** — решения зафиксированы, LLM не переизобретает уже решённое

### DECISIONS.md — архитектурные решения (ADR)

Когда принимается стратегическое решение (выбор технологии, архитектурный подход, модель данных) — записать в `docs/DECISIONS.md`.

Разница с AI_NOTES:
- **AI_NOTES** — тактические, привязаны к блоку задач ("в блоке 2 выбрали Deepgram")
- **DECISIONS.md** — стратегические, живут весь проект ("почему PostgreSQL а не SQLite")

Claude читает DECISIONS.md при каждом старте сессии — чтобы не переизобретать то что уже решили.

Формат:

```markdown
# Архитектурные решения — [Название проекта]

## ADR-001: [Название решения]
**Дата:** YYYY-MM-DD
**Статус:** принято | пересмотрено | отменено
**Контекст:** [Какую проблему решаем]
**Решение:** [Что выбрали]
**Альтернативы:**
- [Вариант A] — почему не подошёл
- [Вариант B] — почему не подошёл
**Последствия:** [Что это значит для проекта]
```

Когда писать ADR:
- Выбор стека / фреймворка / библиотеки
- Архитектурные паттерны (монолит vs микросервисы, REST vs WebSocket)
- Модель данных и выбор БД
- Стратегия деплоя (Docker vs systemd)
- Модель монетизации и платёжная система
- Любое решение которое дорого переделывать

### Делегирование:
**До 10 задач:** работать самому.
**10+ задач:** Delegate mode (Shift+Tab) — координатор + субагенты. Каждый субагент получает: tech-spec + задачи своего блока + AI_NOTES предыдущих блоков.

### UI-итерации через Agentation (для React-проектов)

Если проект использует React — на фазе инфраструктуры (блок 1) установить Agentation:

```bash
npm install agentation agentation-mcp -D
npx agentation-mcp init
```

Добавить в корневой layout (app/layout.tsx или src/App.tsx):
```tsx
import { Agentation } from 'agentation';

// В конце JSX:
{process.env.NODE_ENV === "development" && <Agentation />}
```

Добавить MCP-сервер в проектный `.claude/settings.json`:
```json
{
  "mcpServers": {
    "agentation": {
      "command": "npx",
      "args": ["agentation-mcp", "server"]
    }
  }
}
```

**Workflow UI-итераций (VPS → мак):**
1. На VPS запущен dev-сервер (`npm run dev`) + agentation-mcp
2. С мака проброс портов: `vps-dev <vps-name> <port-dev> <port-mcp>`
3. Открыть `localhost:<port>` в браузере на маке
4. Кликнуть на элемент → написать замечание → Agentation передаёт через MCP
5. В Claude Code: "посмотри мои аннотации" или "поправь фикс #2"
6. Агент правит код → hot-reload → проверяешь в браузере → повторяешь

Без Agentation: «кнопка справа, нет не та, вторая сверху» → 5 итераций.
С Agentation: клик → точный селектор → 1 итерация.
